## Deliverables

### Accepted Order State

`Order.create()` always produces an order in `PENDING` state. PENDING means "accepted for distributed processing" ‚Äî the order has passed authorization and inventory pre-check and is ready for the distributed Saga. PENDING does not mean inventory has been reserved or credit approved; those are Saga steps.

### Why `OrderCreatedDomainEvent` Starts the Saga

`OrderCreatedDomainEvent` is the event that signals an order has been accepted. It carries all information required by the Orchestrator:

- `orderId` ‚Äî the stable aggregate identity for the entire Saga
- `customerId` ‚Äî needed to look up credit limit
- `submittedBy` ‚Äî audit and authorization identity
- `lines` ‚Äî the normalized `List<OrderLine>`, containing exactly the quantities verified against inventory

No `OrderAccepted` event was introduced because there is no semantic distinction here: creation in PENDING state IS acceptance.

### Transaction Boundary

`SubmitCorporateBulkOrderService.execute()` is annotated `@Transactional`. Within a single transaction:

1. `orderRepository.save(order)` ‚Äî writes to `order_service.orders` and `order_service.order_lines`
2. `outboxRepository.save(outboxMapper.map(event, ...))` ‚Äî writes to `platform.outbox_events`

If either step throws, Spring rolls back the entire unit of work. Neither an orphaned order nor an orphaned Outbox event can be produced by a partial execution.

### Result Semantics

`SubmitCorporateBulkOrderResult` returns `orderId` and `status`. The `status` field is `order.status().name()` ‚Äî it is derived directly from the persisted aggregate, not from a parallel variable. The result communicates acceptance for processing, not Saga completion.

### Deferred: Correlation Metadata

`SubmitCorporateBulkOrderCommand` does not carry a `correlationId`. The `DomainEventToOutboxMapper` allocates a new `CorrelationId` on each call. Propagating an inbound request-level correlation into the Outbox event is a follow-up task once a correlation extraction mechanism is established at the REST or security layer.

### New tests

**Test #3** ‚Äî `saga_start_event_payload_contains_normalized_sku_quantities`

Submits two lines for `SKU-001` (10 + 5 = 15). Parses the Outbox event payload with Jackson and asserts `lines[0].sku.value = "SKU-001"` and `lines[0].quantity.value = 15`. Verifies that the Outbox payload carries normalized quantities, not the raw duplicate lines from the command.

**Test #4** ‚Äî `returned_status_equals_persisted_order_status`

Calls `service.execute()`, retrieves the saved order from `InMemoryOrderRepository`, and asserts `result.status() == order.status().name()`. Confirms the result is derived directly from the aggregate, not a hardcoded string.

**Test #5** ‚Äî `outbox_failure_rolls_back_order_persistence` (`OrderSagaStartRollbackIT`)

Real Testcontainers PostgreSQL with Flyway-migrated schema. A `@Primary` `OutboxRepository` bean throws `RuntimeException` on `save()`. Asserts that `order_service.orders` count is unchanged after the failed submission. Verifies that `@Transactional` spans both repositories correctly.

**Test #6** ‚Äî `order_persistence_failure_prevents_outbox_event`

Creates a service with an `OrderRepository` that throws on `save()`. Asserts the exception propagates and `outboxRepository.findAll()` remains empty. Verifies code path: `outboxRepository.save()` is never reached when `orderRepository.save()` throws.

## Architecture Boundaries

Preserved:

```
REST
‚Üí SubmitCorporateBulkOrderService (@Transactional)
‚Üí OrderRepository (save Order)
‚Üí OutboxRepository (save OrderCreatedDomainEvent)

Later:
Outbox relay ‚Üí Kafka ‚Üí Orchestrator
```

The Order Service does not call the Orchestrator, own Saga state, reserve credit, or wait for Saga completion.
