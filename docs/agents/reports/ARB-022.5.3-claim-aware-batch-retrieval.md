# ARB-022.5.3 — Claim-Aware Batch Retrieval

## Context

ARB-022.5.1 introduced `PublishStatus.CLAIMED`, `claimedBy`, and `claimedAt`.
ARB-022.5.2 introduced `claimEvent` for single-event atomic ownership.

The polling runtime still retrieved work via `findPending(limit)`, which returned
rows without establishing ownership. This slice replaces that with a bounded,
claim-aware operation: one call simultaneously selects and claims a batch of
`PENDING` events so only the requesting worker receives them.

---

## Repository API

`OutboxRepository` gains one new operation:

```java
List<OutboxEvent> claimPending(String workerId, Instant claimedAt, int limit);
```

| Constraint | Behaviour |
|---|---|
| `workerId` blank | `IllegalArgumentException` |
| `claimedAt` null | `NullPointerException` |
| `limit < 0` | `IllegalArgumentException` |
| `limit == 0` | returns `List.of()` immediately |
| normal path | returns up to `limit` `CLAIMED` events owned by `workerId` |

The returned list is ordered `occurredAt ASC, eventId ASC` (FIFO with deterministic
UUID tiebreaker).

---

## PostgreSQL Claim Strategy

Two-step within a single `@Transactional` boundary:

### Step 1 — Lock candidate rows

```sql
SELECT id FROM outbox_events
WHERE publish_status = 'PENDING'
ORDER BY occurred_at ASC, id ASC
LIMIT :limit
FOR UPDATE SKIP LOCKED
```

`FOR UPDATE SKIP LOCKED` acquires a row-level exclusive lock on each returned row
and skips any row already held by a concurrent transaction. Two workers executing
this query simultaneously receive disjoint sets of IDs.

### Step 2 — Bulk UPDATE

```jpql
UPDATE OutboxEventEntity e
SET e.publishStatus = 'CLAIMED',
    e.claimedBy     = :workerId,
    e.claimedAt     = :claimedAt
WHERE e.id IN :ids
  AND e.publishStatus = 'PENDING'
```

The `AND e.publishStatus = 'PENDING'` guard prevents a double-claim in the
improbable window where Worker B's Step 1 completes after Worker A's Step 1 but
before Worker A's Step 2. Without the guard, Worker B could overwrite Worker A's
ownership. With it, the late UPDATE affects 0 rows.

---

## Transaction Boundary

`JpaOutboxRepositoryAdapter.claimPending` is annotated `@Transactional`. The
SELECT and UPDATE execute within one short database transaction that commits before
the method returns. No locks are held during Kafka dispatch.

```
short DB transaction
  └── SELECT ... FOR UPDATE SKIP LOCKED  (acquire row locks)
  └── UPDATE ... WHERE status = 'PENDING' (transition + release to CLAIMED)
commit
return claimed events
— later dispatch happens outside this transaction —
```

`claimEvent` (single-event claim from ARB-022.5.2) also received `@Transactional`
as part of this slice: the concurrent test for that method revealed it was missing
from the original implementation.

---

## FIFO Ordering

Primary sort: `occurredAt ASC` — oldest message dispatched first.  
Secondary sort: `eventId ASC` — deterministic UUID tiebreaker when timestamps collide.

Both the native SELECT and the final in-memory sort applied to the loaded entities
use this compound key, so the list returned to the caller is stable regardless of
concurrent writes.

---

## Concurrency Guarantee

Two workers calling `claimPending` concurrently receive disjoint event sets
because:

1. `SKIP LOCKED` ensures their SELECT queries return non-overlapping rows.
2. The `AND status = 'PENDING'` guard in the UPDATE defends against any
   residual race.
3. The `@Transactional` boundary ensures locks are released atomically on commit.

A second `claimPending` call after the first commits will find zero `PENDING`
rows if all events were claimed in the first call.

---

## In-Memory Repository

`InMemoryOutboxRepository.claimPending` is `synchronized` to serialise concurrent
access in multi-threaded test scenarios. It:

1. Identifies candidate IDs in `occurredAt ASC, eventId ASC` order up to `limit`.
2. Iterates the event list to atomically replace matching entries with their
   `CLAIMED` counterparts via `OutboxEvent.claim(workerId, claimedAt)`.
3. Returns a sorted, unmodifiable copy.

In-memory concurrency is coarser than PostgreSQL `SKIP LOCKED` (intrinsic lock
vs. row-level locks), but the observable behaviour is identical for the test
scenarios covered.

## Architectural Decisions

**Why `SELECT ... FOR UPDATE SKIP LOCKED` rather than `UPDATE ... RETURNING`?**

Spring Data JPA's `@Modifying` + JPQL does not support `RETURNING`. A native
`UPDATE ... RETURNING` would work but mixes native SQL with the entity lifecycle.
The two-step approach keeps Step 1 as a native query (which requires native SQL
for `SKIP LOCKED`) and Step 2 as JPQL (which respects the first-level cache and
the `@Modifying` lifecycle). This is the canonical Spring Data JPA pattern for
concurrent claiming.

**Why remove `final` from `JpaOutboxRepositoryAdapter`?**

Spring Boot 4.1 defaults to CGLIB proxying (`spring.aop.proxy-target-class=true`).
CGLIB subclasses the target class to apply `@Transactional` advice; a `final`
class cannot be subclassed. The adapter now depends on Spring AOP for transaction
management, so `final` must be absent.

---

## Limitations

- The `selectPendingIdsForClaim` query is PostgreSQL-specific (native SQL with
  `FOR UPDATE SKIP LOCKED`). H2 does not support this syntax; integration tests
  require a PostgreSQL Testcontainer.
- Claim expiration and abandoned-claim recovery are out of scope (deferred to
  a future slice).
- The polling runtime still calls `findPending(limit)` — migration to
  `claimPending` is deferred to the next slice (ARB-022.5.4 or equivalent).
