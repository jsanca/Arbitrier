# ARB-023.3.1 — Harden Inventory Availability Verification

## Context

The `SubmitCorporateBulkOrderService.verifyInventoryAvailability()` method had four vulnerabilities:

1. **Repeated SKUs caused `IllegalStateException`** — `Collectors.toMap` with no merge function throws on duplicate keys. Commands with multiple lines for the same SKU would fail.
2. **Missing response SKU silently approved** — If the inventory service omitted a requested SKU from its response, the `allMatch` check only examined returned SKUs; absent SKUs were never checked.
3. **Empty response silently approved** — `allMatch` on an empty stream returns `true`, allowing zero-availability through.
4. **No protocol error classification** — All failures were lumped into `ORDER_ITEMS_UNAVAILABLE` (422), even when the inventory service returned malformed responses.

## Summary

Hardened `verifyInventoryAvailability` to: consolidate repeated SKUs before querying; validate response completeness (empty, missing, duplicate, unexpected SKUs); classify protocol violations as `INVENTORY_PROTOCOL_ERROR` (502); preserve insufficient quantity as `ORDER_ITEMS_UNAVAILABLE` (422).

## Deliverables

- `SubmitCorporateBulkOrderService.java` — hardened availability verification
- `OrderProblemCode.java` — added `INVENTORY_PROTOCOL_ERROR`
- `StubInventoryAvailabilityPort.java` — added `StubBehavior` enum for protocol failure simulation
- `SubmitCorporateBulkOrderServiceTest.java` — 14 focused tests

## Architectural Decisions

**Protocol failures vs business failures** — Inventory protocol violations (empty response, malformed response) are classified as upstream service errors (502 Bad Gateway) rather than business failures (422). This distinction matters for error handling: 422 indicates the order cannot be accepted at this time; 502 indicates the inventory service is misbehaving and the request should be retried.

**Consolidation at submission, not at port interface** — SKU consolidation happens in `verifyInventoryAvailability` before building `AvailabilityLineQuery` objects, so the port interface remains unchanged. The production gRPC adapter continues to receive one query per line (with potential duplicates), and the hardening logic validates that the response matches.

## Implementation Notes

**`Integer::sum` merge function** — Changed `Collectors.toMap(keyMapper, valueMapper)` to `Collectors.toMap(keyMapper, valueMapper, Integer::sum)` to sum quantities for duplicate SKUs.

**Response validation order** — Empty check before duplicate check before missing check before unexpected check. Each throws immediately; no cascading validation.

**Stub behavior reset** — `StubInventoryAvailabilityPort.setNextBehavior()` resets to `NORMAL` after one call via `finally` block in `checkAvailability()`. This prevents test pollution between cases.

## Tradeoffs

- **Port interface unchanged** — The `InventoryAvailabilityPort` still returns `List<AvailabilityLineResponse>` per query line. Hardening happens in the application service. This avoids changing the gRPC adapter but means the validation logic is centralized in one caller.
- **Stub supports protocol failure, not the real adapter** — `GrpcInventoryAvailabilityAdapter` does not simulate protocol failures; the stub is for unit tests only.
