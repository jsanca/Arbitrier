# ARB-024.1 — Transport Publication Boundary: Implementation Report

## Context

ARB-023 established the transactional Outbox as the exclusive publication path for the Order submission use case. Deep architecture review (V4 Pro) identified that the Order Service also contained an `OrderEventPublisher` port and a `KafkaOrderEventPublisher` adapter — a direct application-to-Kafka publication path that was never used by the submission service but existed as dead code from an earlier design. ARB-024.1 removes the ambiguity and makes the publication boundary explicit and enforced.

## Publication Boundary (Established)

### The single production publication path

```
SubmitCorporateBulkOrderService  (@Transactional)
  → orderRepository.save(order)
  → outboxRepository.save(outboxMapper.map(event, aggregateId, aggregateType))
  → commit
    ↓
  Outbox Relay (ARB-024.x)
    ↓
  Transport Adapter (JSON dev / Avro+Schema Registry prod)
    ↓
  Kafka
```

### The rejected path (removed)

```
Application Service
  → OrderEventPublisher              ← removed port
    → KafkaOrderEventPublisher       ← removed adapter
      → KafkaTemplate
```

`SubmitCorporateBulkOrderService` never injected `OrderEventPublisher`. Its constructor takes only `OrderRepository`, `OutboxRepository`, `DomainEventToOutboxMapper`, `CustomerAccessPort`, and `InventoryAvailabilityPort`. The dead path was a leftover from before the Outbox refactor that was never cleaned up.

---

## Types Removed

### `OrderEventPublisher` (application/port/outbound)

An application-layer port for direct Kafka publishing. Wrong layer — application ports may only depend on domain types and platform ports, never on Kafka semantics. Nothing in the production submission path injected it.

### `KafkaOrderEventPublisher` (adapter/outbound/kafka)

Direct Kafka publisher that accepted `OrderCreatedDomainEvent` and published synchronously via `KafkaTemplate` before transaction commit — the dual-write anti-pattern the Outbox was designed to prevent.

### `KafkaPublisherConfiguration` (config)

`@ConditionalOnProperty("spring.kafka.bootstrap-servers")` configuration that wired `KafkaTemplate` → `KafkaOrderEventPublisher` → `OrderEventPublisher`. The conditional guard was correctly preventing activation in tests, but the code itself represented an incorrect design intent.

### `KafkaOrderEventPublisherTest`

Unit tests verifying topic routing, partition key, Avro value, and header propagation for the dead adapter.

### `RecordingOrderEventPublisher`

Test double capturing `OrderCreatedDomainEvent` instances for assertion. Wired defensively in 5 persistence test `Config` classes from before the Outbox refactor when `SubmitCorporateBulkOrderService` still injected `OrderEventPublisher`. Since the service no longer injects it, the recording double served no purpose.

---

## Type Retained: `OrderCreatedAvroMapper`

`OrderCreatedAvroMapper` maps `OrderCreatedDomainEvent` → Avro `OrderCreated`. This is a legitimate future Outbox relay transport component:

- Correctly located in `adapter/outbound/kafka` (the relay transport layer)
- Tested by `OrderCreatedAvroMapperTest` (8 tests covering field mapping, metadata, placeholder total)
- Has **no dependency on `KafkaTemplate`** or any removed types
- Does not belong in any application service

Changes applied:
- Removed `@Component` — the mapper should not be auto-scanned as a Spring bean until the Avro relay configuration wires it explicitly (ARB-024.x)
- Updated Javadoc: "Outbox relay transport component (ARB-024); application services must not reference this class"

## Architecture Enforcement

### New ArchUnit rule

```java
@Test
void application_must_not_depend_on_kafka_publisher_adapters() {
    ArchRuleDefinition.noClasses().that().resideInAPackage("..application..")
            .should().dependOnClassesThat().resideInAPackage("..adapter.outbound.kafka..")
            .because("application services publish through OutboxRepository; " +
                     "transport publication adapters belong in the adapter layer")
            .allowEmptyShould(true)
            .check(classes);
}
```

This supplements the existing `application_must_not_depend_on_avro_or_kafka` rule (which blocks `KafkaTemplate` etc.) with an explicit adapter-boundary rule that prevents any future class in `..application..` from acquiring a reference to any class in `..adapter.outbound.kafka..` regardless of whether that class imports Kafka directly.

### Existing rules that already enforced this

| Rule | What it prevents |
|---|---|
| `application_must_not_depend_on_adapter` | No `..application..` class may depend on `..adapter..` classes |
| `application_must_not_depend_on_avro_or_kafka` | No Kafka, Avro, or Spring Kafka imports in application layer |

The new rule makes the publication boundary intent explicit with a `because()` clause documenting the architectural reason.

## JSON Dev / Avro Prod Adapter Strategy (Documented)

Future ARB-024 slices will implement two alternative transport adapters behind the same relay boundary:

```
Outbox Relay
  → (dev)  JsonTransportAdapter   → Kafka (raw JSON)
  → (prod) AvroTransportAdapter   → Schema Registry → Kafka (Avro)
```

These are **alternative adapters**, not parallel publications. At runtime exactly one adapter is active (selected by configuration). `OrderCreatedAvroMapper` is the Avro adapter's mapping component. A `JsonTransportAdapter` would publish the outbox `payload` field directly (already JSON from `JacksonEventSerializer`).

## Stable Event Identity (Follow-up: ARB-024.2)

`DomainEventToOutboxMapper` derives the outbox `eventType` from `event.getClass().getSimpleName()` — currently `"OrderCreatedDomainEvent"`. This is a Java class name, not a stable logical event type.

Before Schema Registry subjects are defined, a dedicated follow-up task must establish:

- **Stable logical event type** — e.g., `"order.created"` or `"OrderCreated"`, independent of Java class name
- **Event version** — e.g., `"v1"`, separate from schema version in the Avro metadata
- **Subject naming** — e.g., `arbitrier.order.OrderCreated-value` (Schema Registry subject naming strategy)
- **Independence from Java class names** — renaming `OrderCreatedDomainEvent` must not silently change the published event type

This is **not implemented in ARB-024.1** per task scope. It is tracked as an open question for ARB-024.2.

## Transactional Safety

`SubmitCorporateBulkOrderService` constructor was and remains:

```java
public SubmitCorporateBulkOrderService(
    OrderRepository orderRepository,
    OutboxRepository outboxRepository,
    DomainEventToOutboxMapper outboxMapper,
    CustomerAccessPort customerAccessPort,
    InventoryAvailabilityPort inventoryAvailabilityPort)
```

No `OrderEventPublisher` parameter was ever present in the current production code. The `@Transactional` boundary covers exactly `orderRepository.save()` + `outboxRepository.save()` + commit. Nothing else.
