# ARB-022.2.2 — Kafka Outbound Message Publisher Adapter

## Context

ARB-022.2.1 introduced `OutboundMessagePublisher.publish(OutboxEvent)` as the runtime-neutral port.
This slice implements that port using Spring Kafka.

---

## Adapter Package and Bean Wiring

**Adapter class:** `com.arbitrier.platform.messaging.kafka.outbound.KafkaOutboundMessagePublisher`

**Auto-configuration:** `com.arbitrier.platform.spring.KafkaPublisherAutoConfiguration`

Activation conditions (both required):
- `@ConditionalOnClass(KafkaTemplate.class)` — `spring-kafka` is on the classpath
- `@ConditionalOnProperty("spring.kafka.bootstrap-servers")` — a broker is configured

The auto-configuration creates named beans `outboundMessageProducerFactory` and `outboundMessageKafkaTemplate` (both `@ConditionalOnMissingBean(name = "…")`) and a `kafkaOutboundMessagePublisher` bean (`@ConditionalOnMissingBean(OutboundMessagePublisher.class)`), so services can override any tier.

`KafkaPublisherAutoConfiguration` is registered in `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`.

---

## KafkaTemplate Generic Types

`KafkaTemplate<String, String>` — key and value are both UTF-8 strings.

The outbox payload is already serialized (JSON string). Passing it as-is to a `String`-value template avoids any deserialization/re-serialization. Avro serialization with Schema Registry is deferred to a later slice.

A dedicated `outboundMessageProducerFactory` is created with `StringSerializer` for both key and value. This named factory avoids conflict with `KafkaTemplate<String, Object>` used by `order-service`'s Avro Kafka publisher.

---

## Key Choice

`aggregateId` is used as the Kafka partition key.

Rationale: all messages (events and commands) for the same aggregate go to the same Kafka partition, preserving ordering without inspecting message content. `eventId` would also be unique but would scatter messages for the same aggregate across partitions.

---

## Destination Resolution

`OutboundRoutingStrategy.resolveDestination(OutboxEvent)` is called before `KafkaTemplate.send()`. The topic string returned is used directly as the Kafka topic name. No topic-name logic is embedded in the adapter.

## Asynchronous Acknowledgement Limitation

`KafkaTemplate.send()` is non-blocking. `publish()` returns `void` immediately after initiating the send. The adapter does **not** call `OutboxRepository.markPublished()` or `markFailed()`.

**OPEN QUESTION:** The `void` return type on `OutboundMessagePublisher` makes it impossible for the adapter to notify the outbox drainer of delivery success or failure on the same call path. The future drainer slice must resolve this — either by: (a) polling the returned `CompletableFuture` from inside the drainer, (b) registering a callback on the future before handing control back, or (c) changing the port return type to `CompletableFuture<Void>`. Changing the port return type is out of scope for this slice.

---

## Pre-existing Publisher Note

`KafkaOrderEventPublisher` in `order-service` is a service-specific Avro publisher for `OrderCreatedDomainEvent`. It predates the outbox model and is not affected by this slice. It publishes directly to Kafka (not via the outbox) and is a candidate for consolidation in a future cleanup task.

## Open Questions

1. **Acknowledgement coordination** — `void` port return type prevents the adapter from signalling delivery outcome to the drainer. See Asynchronous Acknowledgement Limitation above.
2. **`OutboundRoutingStrategy` bean** — services must provide an `OutboundRoutingStrategy` bean for `KafkaPublisherAutoConfiguration` to create the `OutboundMessagePublisher`. No default implementation exists yet; the auto-configuration will fail to wire the publisher if no strategy is present.
