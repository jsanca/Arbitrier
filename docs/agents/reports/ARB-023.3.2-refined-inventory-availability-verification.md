# ARB-023.3.2 — Refine Inventory Availability Verification Responsibilities

## Context

ARB-023.3.1 hardened the inventory availability verification by adding protocol validation to `SubmitCorporateBulkOrderService`. Architecture review identified that this design placed protocol-awareness into the business use case, violating the principle that adapters handle transport/protocol concerns while application services handle business decisions.

## Summary

Refactored `SubmitCorporateBulkOrderService` to own only the business question "is there enough inventory?". Protocol validation (missing SKU, duplicate SKU, unexpected SKU, malformed response) belongs to the outbound adapter. Added explicit `aggregateRequestedQuantitiesBySku()` helper. Ensured the same normalized quantities are used for both validation and Order persistence.

## Architectural Refinements

### 1. Protocol validation moved to adapter layer

`validateInventoryResponse()` was removed from `SubmitCorporateBulkOrderService`. Protocol violations (empty response, missing SKU, duplicate SKU, unexpected SKU) are the adapter's responsibility. The adapter throws `InventoryAvailabilityProtocolException` (or other `InventoryAvailabilityException` subtypes) which propagate up through the application service unchanged.

**Rationale:** The application service should not be aware of how the inventory service communicates (gRPC, HTTP, in-memory). If the transport changes, the application service should not need to change.

### 2. Explicit SKU aggregation helper

Added `aggregateRequestedQuantitiesBySku(SubmitCorporateBulkOrderCommand)` returning `LinkedHashMap<String, Integer>` with insertion-order-preserving semantics. Uses `Collectors.toMap(...Integer::sum, LinkedHashMap::new)` to merge quantities for duplicate SKUs.

**Name:** `aggregateRequestedQuantitiesBySku` — explicit about the operation.

### 3. Single source of truth for validated quantities

Before this refinement, `verifyInventoryAvailability()` aggregated SKUs but `Order.create()` used `command.lines()` directly. This meant validation and persistence could operate on different representations.

After: both `verifyInventoryAvailability()` and `Order.create()` use the same `LinkedHashMap<String, Integer>` produced by `aggregateRequestedQuantitiesBySku()`.

### 4. Quantity validation already enforced

`SubmitCorporateBulkOrderLineCommand` already enforces `quantity > 0` in its constructor via `Require.isTrue`. `Quantity` value object also validates `value > 0`. No changes needed.

### 5. INVENTORY_PROTOCOL_ERROR removed

`OrderProblemCode.INVENTORY_PROTOCOL_ERROR` was removed. Protocol failures propagate as `InventoryAvailabilityProtocolException` from the adapter, not as application-level problem codes.

## Deliverables

## Tradeoffs

- **Protocol validation now in adapter** — if the gRPC adapter doesn't validate response completeness (missing/duplicate/unexpected SKUs), those violations go undetected. The existing `InventoryAvailabilityGrpcResponseMapper` validates `UNSPECIFIED` status and `AVAILABLE` with non-empty unavailable_items, but does not validate response SKU completeness against query SKUs. This could be added to the mapper or adapter in a follow-up if required.
- **Protocol exceptions propagate as-is** — `InventoryAvailabilityProtocolException` thrown by the adapter propagates through the application service without translation. This means callers receive a 500 error (unhandled exception) rather than a structured `ApplicationProblemException`. The alternative (catching at service and converting to `ApplicationProblemException`) would require modifying the application service to depend on adapter exception types, which couples the service to the transport.

## Open Questions

- Should `InventoryAvailabilityGrpcResponseMapper` or `GrpcInventoryAvailabilityAdapter` validate that the response contains exactly the requested SKUs (no missing, no duplicates, no unexpected)? This would require access to the original query SKUs in the mapper, which is already passed in.
- Should `InventoryAvailabilityProtocolException` be caught in the application service and converted to a structured error response with a defined problem code?

## Follow-ups

- Add protocol completeness validation to `InventoryAvailabilityGrpcResponseMapper` or `GrpcInventoryAvailabilityAdapter` (validate response SKUs match query SKUs)
- Consider whether application service should catch `InventoryAvailabilityException` and convert to structured `ApplicationProblemException` with a defined problem code
