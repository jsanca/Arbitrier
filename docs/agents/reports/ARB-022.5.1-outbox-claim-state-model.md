# ARB-022.5.1 — Outbox Claim State Model

## New Status: CLAIMED

`PublishStatus` now has four values:

| Value | Meaning |
|-------|---------|
| `PENDING` | Awaiting dispatch; eligible for polling |
| `CLAIMED` | Temporarily owned by one worker during dispatch |
| `PUBLISHED` | Successfully published; terminal |
| `FAILED` | Publishing failed; terminal |

`CLAIMED` sits between `PENDING` and the terminal states. It represents the window in which a worker has taken responsibility for a message but has not yet confirmed success or failure.

## Claim Metadata

Two new fields on `OutboxEvent`:

| Field | Type | Meaning |
|-------|------|---------|
| `claimedBy` | `String` | Worker identifier that owns the claim |
| `claimedAt` | `Instant` | Timestamp when the claim was recorded |

Both fields are nullable. They are populated when status is `CLAIMED` and are `null` for every other status.

## Lifecycle Transitions

```
PENDING ──claim(workerId, claimedAt)──▶ CLAIMED ──markPublished(publishedAt)──▶ PUBLISHED
                                                  └──markFailed(failedAt)──────▶ FAILED
```

All three transitions are domain methods on `OutboxEvent` (which is a Java record) and return a new instance:

- **`claim(workerId, claimedAt)`** — only legal from `PENDING`; throws `IllegalStateException` if status is anything else; validates `workerId` non-blank and `claimedAt` non-null before constructing the new record.
- **`markPublished(publishedAt)`** — only legal from `CLAIMED`; throws `IllegalStateException` otherwise; clears `claimedBy` and `claimedAt` on the new record.
- **`markFailed(failedAt)`** — only legal from `CLAIMED`; throws `IllegalStateException` otherwise; increments `attemptCount`, sets `lastAttempt = failedAt`, clears `claimedBy` and `claimedAt`.

State-machine validation uses `IllegalStateException` (invalid source state) while argument validation uses `IllegalArgumentException` / `NullPointerException` (bad input). These are distinct categories.

## Constructor Invariants

The `OutboxEvent` compact constructor enforces claim-field consistency:

- If `publishStatus == CLAIMED`: `claimedBy` must be non-blank and `claimedAt` must be non-null.
- If `publishStatus != CLAIMED`: both `claimedBy` and `claimedAt` must be `null`.

This means an `OutboxEvent` can never be in an inconsistent state regardless of how it was created.

## Schema Changes (Flyway V4)

```sql
ALTER TABLE outbox_events ADD COLUMN claimed_by VARCHAR(255);
ALTER TABLE outbox_events ADD COLUMN claimed_at TIMESTAMPTZ;
ALTER TABLE outbox_events DROP CONSTRAINT outbox_publish_status_chk;
ALTER TABLE outbox_events ADD CONSTRAINT outbox_publish_status_chk
    CHECK (publish_status IN ('PENDING', 'CLAIMED', 'PUBLISHED', 'FAILED'));
```

Both columns are nullable. The existing `CHECK` constraint is replaced to include `'CLAIMED'`. Existing `PENDING`, `PUBLISHED`, and `FAILED` rows are unaffected.

## JPA Mapping

`OutboxEventEntity` has two new columns mapped with `@Column`:

```java
@Column(name = "claimed_by")
private String claimedBy;

@Column(name = "claimed_at")
private Instant claimedAt;
```

`JpaOutboxRepositoryAdapter`:
- `toEntity()` maps both fields from `OutboxEvent`.
- `toRecord()` reads both fields from `OutboxEventEntity`.
- `markPublished()` and `markFailed()` set both columns to `null` before saving.

## findPending Behaviour

Both `findPending()` and `findPending(limit)` filter on `publish_status = 'PENDING'`. `CLAIMED` events are excluded. This is consistent across:
- `JpaOutboxRepositoryAdapter` (SQL via Spring Data query method)
- `InMemoryOutboxRepository` (Java stream filter)

## Compatibility Decisions

- `OutboxRepository.markPublished(UUID)` and `markFailed(UUID)` operate on the entity directly and accept any source status. This preserves backward compatibility with existing callers that transition directly from `PENDING` rather than via `CLAIMED`. The domain methods (`OutboxEvent.markPublished`, `OutboxEvent.markFailed`) are stricter — they require `CLAIMED`.
- All existing call sites for the 14-argument `OutboxEvent` constructor were updated to 16 arguments (`claimedBy=null`, `claimedAt=null`). No service modules create `OutboxEvent` directly; all creation paths go through `DomainEventToOutboxMapper`.

## Intentionally Deferred

The following are **out of scope** for this slice and must not be implemented until their own tasks are defined:

- Atomic claim operation (`SELECT FOR UPDATE SKIP LOCKED` or equivalent)
- Batch claiming
- Worker ID generation (hostname, pod name)
- Claim expiration and abandoned-claim recovery
- Multiple concurrent workers
- Claim query (find messages claimable by a specific worker)
- DEAD status
- Retry counters and backoff scheduling
- Metrics and tracing for claim lifecycle
