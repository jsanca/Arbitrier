# ARB-023.4 — REST Order Entry Point: Implementation Report

## Context

ARB-023.3 delivered the `GrpcInventoryAvailabilityAdapter` behind the `InventoryAvailabilityPort` outbound port. ARB-023.4 wires that port into the order creation flow so availability is verified before any order is persisted or any domain event is published.

The REST endpoint and controller existed from ARB-007 and ARB-010. This task extended the application layer, added the HTTP exception mapping, and added the tests required to prove the full path.

---

## Summary

- `InventoryAvailabilityPort` is now called inside `SubmitCorporateBulkOrderService.execute()` before `Order.create()`, before `orderRepository.save()`, and before `outboxRepository.save()`.
- `ORDER_ITEMS_UNAVAILABLE` (422) added to `OrderProblemCode`.
- `OrderInventoryExceptionHandler` added in `adapter/inbound/rest/` to map the four typed gRPC transport exceptions to 504 / 503 / 500.
- 27 service unit tests, 12 controller tests, and 3 integration tests all pass. Full suite: **147 tests, 0 failures**.

## Final Call Path

```
POST /api/orders
    ↓
SubmitCorporateBulkOrderController
    extracts JWT subject → submittedByUserId
    maps body + userId → SubmitCorporateBulkOrderCommand
    ↓
SubmitCorporateBulkOrderService.execute()   [@Transactional]
    1. customerAccessPort.canSubmitOrder()      → 403 if denied
    2. verifyInventoryAvailability()            → 422 if any line insufficient
                                                → 504/503/500 if gRPC fails
    3. Order.create()
    4. orderRepository.save()
    5. outboxRepository.save(mapper.map(event))
    ↓
SubmitCorporateBulkOrderResult → 201 Created
```

## HTTP Status Mapping

| Condition | Exception | HTTP |
|---|---|---|
| Valid, sufficient inventory | — | 201 Created |
| Malformed request | `IllegalArgumentException` (via command ctor) | 400 Bad Request |
| Missing JWT | Spring Security | 401 Unauthorized |
| Customer access denied | `ApplicationProblemException(CUSTOMER_ACCESS_DENIED)` | 403 Forbidden |
| Insufficient inventory | `ApplicationProblemException(ORDER_ITEMS_UNAVAILABLE)` | 422 Unprocessable Entity |
| Inventory gRPC timeout | `InventoryAvailabilityTimeoutException` | 504 Gateway Timeout |
| Inventory service down | `InventoryAvailabilityRemoteUnavailableException` | 503 Service Unavailable |
| Protocol/internal gRPC error | `InventoryAvailabilityProtocol/InternalException` | 500 Internal Server Error |

## Invariant: Unavailability Prevents All Side Effects

Verified by `insufficient_stock_does_not_save_order_or_event`, `availability_is_checked_before_order_is_created`, `inventory_timeout_propagates_without_saving_order`, `inventory_remote_unavailable_propagates_without_saving_order` in `SubmitCorporateBulkOrderServiceTest`, and `insufficient_inventory_returns_422_and_no_order_saved` in `OrderCreationIT`:

```
Inventory unavailable ⇒ no Order saved ⇒ no OutboxEvent ⇒ no Saga
```

All three failure paths (business unavailability, gRPC timeout, gRPC remote failure) roll back the `@Transactional` boundary and leave both repositories empty.

---

## Architectural Decisions

**`OrderInventoryExceptionHandler` in `adapter/inbound/rest/`**: `PlatformExceptionHandler` handles `ApplicationProblemException` generically. The gRPC transport exceptions (`InventoryAvailability*Exception`) are order-service-specific and must not be known by the platform module. A second `@RestControllerAdvice` in `order-service` handles them without polluting the platform.

**Availability check in the application service, not the controller**: The controller has no business knowledge. `InventoryAvailabilityPort` is called inside `execute()` so the `@Transactional` boundary covers both the check and any subsequent state mutations atomically.

**`verifyInventoryAvailability()` placement**: Called after the customer access check, before `OrderId` allocation. An access-denied failure is cheaper than an availability call, so the gate order avoids unnecessary gRPC round-trips for unauthorized requests.

## Open Questions

- `correlationId` is not yet part of `SubmitCorporateBulkOrderCommand` — noted as `OPEN QUESTION` in the service log. Observability wiring deferred.
- `ORDER_ITEMS_UNAVAILABLE` currently returns the set of unavailable SKUs at a summary level only. Line-level detail (which lines, how many available) is not yet in the API response body — consistent with the existing `ProblemResponse` envelope which has `code` and `message` only. A dedicated error extension field would require an API contract change.
