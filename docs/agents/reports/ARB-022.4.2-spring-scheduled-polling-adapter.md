# ARB-022.4.2 — Spring Scheduled Polling Adapter

## Class and Package

**Scheduler:** `OutboxPollingScheduler`  
**Package:** `com.arbitrier.platform.messaging.outbox.spring`

**Auto-configuration:** `OutboxSchedulingAutoConfiguration`  
**Package:** `com.arbitrier.platform.spring`

## Scheduled Method

```java
@Scheduled(
    fixedDelayString  = "${arbitrier.messaging.outbox.polling.fixed-delay-ms:10000}",
    initialDelayString = "${arbitrier.messaging.outbox.polling.initial-delay-ms:5000}"
)
public void poll() { ... }
```

The method calls `pollingService.pollOnce()` and returns immediately. The returned `CompletionStage<Void>` is not joined — no blocking.

## Fixed-Delay Decision

Fixed delay (`fixedDelayString`) is used rather than fixed rate (`fixedRate`). The interval is measured from the completion of the previous cycle, not from its start.

**Rationale:** `OutboxPollingService` already prevents overlapping cycles via `AtomicBoolean`. With fixed rate, Spring would queue up additional ticks during a slow cycle, producing a burst of immediately-skipped invocations once the cycle finishes. Fixed delay eliminates that noise — the next tick begins only after the current cycle resolves.

## Default Values

| Property | Default |
|----------|---------|
| `arbitrier.messaging.outbox.polling.initial-delay-ms` | 5 000 ms |
| `arbitrier.messaging.outbox.polling.fixed-delay-ms` | 10 000 ms |

Both are overridable in `application.yml` without changing any Java code.

## Failure Observation

Asynchronous failures are observed via `whenComplete`:

```java
pollingService.pollOnce()
    .whenComplete((ignored, failure) -> {
        if (failure != null) {
            log.error("Outbox polling cycle failed asynchronously: {}",
                    failure.getMessage(), failure);
        }
    });
```

Synchronous throws from `pollOnce()` are caught at the scheduler boundary:

```java
} catch (RuntimeException ex) {
    log.error("Outbox polling cycle failed immediately: {}", ex.getMessage(), ex);
}
```

Neither path rethrows — future scheduled ticks continue in all failure scenarios.

## Activation Conditions

`OutboxSchedulingAutoConfiguration` carries:

```java
@ConditionalOnBean(OutboxPollingService.class)
```

The scheduler is registered only when the full outbox runtime (service, repository, dispatcher, publisher) has been wired and an `OutboxPollingService` bean exists. No fake or no-op polling service is created.

## Scheduling Enablement

`@EnableScheduling` was not declared anywhere in the codebase before this slice. It is placed on `OutboxSchedulingAutoConfiguration`, which is itself conditional on `OutboxPollingService`. This ties scheduling activation to the outbox runtime — contexts that never start the outbox pipeline do not have scheduling enabled.

`OutboxSchedulingAutoConfiguration` is registered in:  
`META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`

## ArchUnit Safety

The `messaging_domain_must_not_depend_on_jpa_or_spring` ArchUnit rule restricts only the exact package `com.arbitrier.platform.messaging.outbox`. The new package `com.arbitrier.platform.messaging.outbox.spring` is a subpackage and is not covered by the exact-match rule, so Spring dependencies in `OutboxPollingScheduler` do not violate the existing architecture constraints.
