# ARB-023.1 — Inventory Availability gRPC Contract

## Context

ARB-023 introduces the first executable business entry flow in Arbitrier. Order Service
must query Inventory Service synchronously to determine whether requested products are
currently in stock before accepting an order and starting its Saga.

This slice defines the wire contract for that query. No server or client adapters are
implemented here — only the transport-neutral protobuf definition and generated Java types.

---

## Summary

One `.proto` file defines the full gRPC contract for the availability query. The build now
generates 12 Java classes (message types + gRPC stubs) from that file. 14 new contract
tests verify generated types, round-trip serialization, and descriptor assertions.

The contracts module retains the pre-existing Avro contract (20 Avro tests still pass).

## Contract Decisions

### Java package: `com.arbitrier.contracts.inventory.v1`

The task baseline used `io.arbitrier.contracts.inventory.v1`. The existing Avro convention
uses `com.arbitrier.contracts.*`. Adjusted to `com.arbitrier.contracts.inventory.v1` to
keep all contract-generated types under the same root package, making it easier for
services to import both Avro and gRPC types from a single known prefix.

### Separate `v1` sub-package

Avro contracts use `com.arbitrier.contracts.inventory` without versioning. The gRPC
contract adds a `v1` layer (`com.arbitrier.contracts.inventory.v1`) because gRPC
contracts are versioned at the package level per protobuf compatibility rules. This
also avoids any name collision with Avro-generated inventory types in the future.

### os-maven-plugin as build extension, not plugin

`os-maven-plugin` must be declared in `<build><extensions>` (not `<plugins>`) so that
its property resolution (`${os.detected.classifier}`) runs before the protobuf plugin
configuration is evaluated.

### `javax.annotation-api:1.3.2`

gRPC-generated stubs use `@javax.annotation.Generated`, which is not in the JDK module
system since Java 9. Spring Boot 4.x uses the `jakarta.*` namespace and does not manage
this artifact. Added explicitly to the contracts dependency list at compile scope.

### Versions chosen

| Artifact | Version | Reason |
|---|---|---|
| `protobuf-java` | 3.25.5 | Latest 3.x stable; broadly compatible |
| `grpc-java` | 1.65.0 | Latest stable release at time of implementation |
| `os-maven-plugin` | 1.7.1 | Latest stable; supports macOS arm64 and Linux x86_64 |
| `protobuf-maven-plugin` | 0.6.1 | Well-established; pairs with os-maven-plugin |

## Compatibility Considerations

The proto file documents protobuf compatibility rules inline:
- Never reuse field numbers
- Never renumber existing fields
- Reserve removed field numbers and names
- Preserve enum zero values as `UNSPECIFIED`

Adding `UnavailabilityReason.UNAVAILABILITY_REASON_PRODUCT_NOT_FOUND` allows the server
adapter (ARB-023.2) to distinguish missing products from insufficient stock without
callers needing a semantic change — backward-compatible.

---

## Explicit Confirmations

**Reservation is out of scope.** The proto file is annotated throughout to clarify
this is a read-only precondition query. The `CheckAvailability` RPC does not reserve,
allocate, decrement, lock, or mutate inventory. The comment block on the proto service
explicitly documents the AVAILABLE → stock consumed → Saga REJECTED sequence as expected
behavior, not a contract inconsistency.

**No commit was made.** Changes are uncommitted per task policy.

## Open Questions

- **Native Image:** `protobuf-java:3.25.5` accesses `sun.misc.Unsafe::arrayBaseOffset` at
  runtime (logged as a JDK warning). GraalVM native compilation will require a
  `RuntimeHintsRegistrar` or reflection config for the generated proto classes. Deferred
  to the native image task (ADR-0007).

- **Protobuf 4.x migration:** `protobuf-java` 4.x removes the `sun.misc.Unsafe` usage.
  Upgrading from 3.25.x to 4.x is a breaking wire-compatibility change and requires a
  full contract migration. Out of scope for this slice.
