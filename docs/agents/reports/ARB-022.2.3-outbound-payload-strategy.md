# ARB-022.2.3 — Outbound Payload Strategy

## Rationale

`KafkaOutboundMessagePublisher` (ARB-022.2.2) passed `OutboxEvent.payload()` directly as the Kafka record value. This worked for the current JSON development format but would require modifying the publisher whenever the transport format changes (e.g. switching to Avro). Introducing an explicit serialization strategy decouples format decisions from transport mechanics.

## Abstraction

### `OutboundPayloadSerializer`

**Package:** `com.arbitrier.platform.messaging.outbox`

```java
public interface OutboundPayloadSerializer {
    String serialize(OutboxEvent event);
}
```

The interface lives in the outbox contract package alongside `OutboundRoutingStrategy` and `OutboundMessagePublisher`. It is transport-neutral: no Kafka or Avro types appear in the interface or its contract. The returned `String` is used as-is by the current `KafkaTemplate<String, String>` publisher adapter.

## Current JSON Implementation

### `JsonOutboundPayloadSerializer`

**Package:** `com.arbitrier.platform.messaging.serialization`

Returns `OutboxEvent.payload()` unchanged. The outbox pipeline already stores the event as a serialized JSON string (via `EventSerializer` / `JacksonEventSerializer`). This implementation surfaces that stored payload for transmission — no re-serialization occurs. Behavior is identical to the pre-ARB-022.2.3 publisher; the abstraction boundary is the only addition.

## Kafka Publisher Update

`KafkaOutboundMessagePublisher` now accepts `OutboundPayloadSerializer` as a third constructor argument. The publish flow is:

```
OutboxEvent
↓ routingStrategy.resolveDestination(message)   → topic
↓ payloadSerializer.serialize(message)           → payload string
↓ new ProducerRecord(topic, aggregateId, payload)
↓ kafkaTemplate.send(record)
```

## Auto-Configuration Update

`KafkaPublisherAutoConfiguration` registers a `@ConditionalOnMissingBean(OutboundPayloadSerializer.class)` bean backed by `JsonOutboundPayloadSerializer`. Services that need Avro serialization can override this by providing their own `OutboundPayloadSerializer` bean without touching the publisher or auto-configuration.

## Why Avro Is Intentionally Deferred

- Schema Registry integration requires a dedicated ADR (registry URL, subject naming strategy, compatibility mode).
- `KafkaTemplate<String, String>` would need to become `KafkaTemplate<String, byte[]>` or use an Avro-specific template; that is a separate publisher adapter concern.
- The interface return type (`String`) is sufficient for JSON. A future Avro interface variant may change the return type or introduce a generic parameter — that refactor is bounded by the clear extension point introduced here.

## Future Extension Points

1. An `AvroOutboundPayloadSerializer` would re-encode the stored JSON payload as Avro binary using a `SchemaRegistryClient`. The publisher adapter would also change to `KafkaTemplate<String, byte[]>` or a typed Avro template.
2. Runtime profile switching (JSON vs. Avro) can be wired at auto-configuration time via `@ConditionalOnProperty` without modifying either the publisher or the serializer interface.
3. The interface and its placement in `messaging.outbox` are protected by the existing `messaging_contracts_must_not_depend_on_kafka` ArchUnit rule.
