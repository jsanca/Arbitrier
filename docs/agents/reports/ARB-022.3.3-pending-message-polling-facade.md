# ARB-022.3.3 — Pending Message Polling Facade

## Service

**Class:** `OutboxPollingService`  
**Package:** `com.arbitrier.platform.messaging.outbox.application`

Dependencies:
- `SequentialPendingDispatchService` — `dispatchPending(int limit)`

No Kafka. No repository dependency. No Spring annotations.

## What One Polling Cycle Means

A single call to `pollOnce()` retrieves at most `batchSize` pending messages and dispatches them sequentially. It represents one bounded, on-demand cycle — not a loop, not a recurring job.

The class owns one responsibility: running one cycle when told to. Scheduling, fixed-rate execution, overlap prevention, and distributed coordination all belong to later slices.

## Public API

```java
CompletionStage<Void> pollOnce()
```

Delegates directly to:

```java
sequentialDispatch.dispatchPending(batchSize)
```

Returns the same `CompletionStage` the delegate returns — completion and failure semantics are preserved unchanged.

## Batch-Size Validation

`batchSize` is a constructor parameter:

```java
public OutboxPollingService(SequentialPendingDispatchService sequentialDispatch, int batchSize) {
    this.sequentialDispatch = Require.notNull(sequentialDispatch, "sequentialDispatch");
    if (batchSize <= 0) {
        throw new IllegalArgumentException("batchSize must be positive: " + batchSize);
    }
    this.batchSize = batchSize;
}
```

Zero is rejected — a polling facade with a zero batch size has no useful behavior. Negative values are equally rejected. A future auto-configuration slice may bind `arbitrier.messaging.outbox.polling.batch-size` to this constructor without changing the class.

## Delegation Flow

```
pollOnce()
  → sequentialDispatch.dispatchPending(batchSize)
      → OutboxRepository.findPending(batchSize)   [up to batchSize PENDING events]
      → DispatchOutboxMessageService.dispatch(e)  [each, sequentially]
      → CompletionStage<Void>
```

## Failure Semantics

| Scenario | Behavior |
|----------|----------|
| Delegate completes normally | `pollOnce()` completes normally |
| Delegate returns exceptional stage | Stage propagates unchanged |
| Delegate throws synchronously | Exception propagates unchanged |

No failures are caught or converted. The caller (future scheduler) decides how to log, isolate, or retry a failed cycle.

## Why Scheduling Is Deferred

This class represents behavior, not timing. Scheduling concerns — fixed delay, initial delay, overlap detection, distributed leases — belong to a distinct runtime configuration slice. Keeping them separate allows the polling logic to be unit-tested without any timer, thread, or scheduler dependency.
