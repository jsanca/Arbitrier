# ARB-023.2A — Inventory Stock Availability Persistence

## Context

ARB-023.2 left `InventoryAvailabilityQueryPort` without a production implementation — the only
concrete availability source was an in-memory test adapter. ARB-023.2A closes that gap by
providing a persistent stock model, a production JPA adapter, and a full gRPC-to-PostgreSQL
integration proof.

---

## Summary

A production `InventoryAvailabilityQueryPort` implementation backed by PostgreSQL was added to
`inventory-service`. The implementation covers: a Flyway migration (V5), a JPA entity, a Spring
Data repository, a two-query availability adapter with reservation deduction, corrected gRPC
exception mapping, and a complete set of unit, persistence, and gRPC-to-database integration
tests. 38 new tests pass; all 131 existing tests continue to pass.

## Preliminary Model Findings

| Question | Finding |
|---|---|
| Meaning of `stock_reservations` | Aggregate that tracks whether a warehouse fulfilled a reservation request — states: RESERVED, PARTIALLY_RESERVED, REJECTED, RELEASED |
| Meaning of `stock_reservation_lines` | Per-SKU demand within a reservation — `sku_code`, `requested_quantity` |
| Meaning of `stock_allocations` | Per-warehouse fulfillment against a line — `warehouse_id`, `sku`, `quantity` |
| Active reservation states | **RESERVED**, **PARTIALLY_RESERVED** consume availability; REJECTED and RELEASED do not |
| When a reservation begins consuming availability | On creation with status RESERVED or PARTIALLY_RESERVED |
| When a reservation stops consuming availability | On transition to REJECTED or RELEASED |
| Stock global vs. warehouse-specific | `inventory_stock` is global per SKU (no warehouse column). Warehouse breakdown lives in `stock_allocations`. No warehouse_id added speculatively |
| SKU identity | String `product_id` / `sku_code` — no `ProductId` domain type exists |
| Existing stock abstraction | None — the schema had only reservation tables before this task |
| Existing ports duplicating behavior | `WarehouseAllocationPort` and `StockAvailabilityPort` exist but have no production implementations; `InventoryAvailabilityQueryPort` is the correct port for this task |

## Stock Source-of-Truth Decision

Introduced `inventory_service.inventory_stock` as the authoritative on-hand stock source.
No `reserved_quantity` column is stored in this table — it would duplicate the authoritative
data already in `stock_reservations` / `stock_allocations` and could drift.

---

## Active Reservation States

States that consume availability (deducted from on-hand quantity):

```
RESERVED
PARTIALLY_RESERVED
```

States that do not consume availability (excluded from deduction):

```
REJECTED
RELEASED
```

This is enforced as a SQL `IN` filter in the native reservation query. The constant is defined
in `JpaInventoryAvailabilityQueryAdapter.ACTIVE_STATUSES`.

---

## Availability Formula

```
availableQuantity = max(0, on_hand_quantity - SUM(active allocation quantities))
```

If `SUM(active allocation quantities)` exceeds `on_hand_quantity` due to inconsistent data,
available quantity is clamped to 0 and a WARN log entry is emitted. Negative values never
reach the gRPC response.

## Migration Details

**File:** `V5__create_inventory_stock_table.sql`

**Version:** V5 — chosen because the shared Flyway namespace (platform + inventory_service
locations) already contains V2, V3, V4 from the platform module. Using V5 avoids the
`Found more than one migration with version N` error.

```sql
CREATE TABLE inventory_service.inventory_stock (
    product_id          VARCHAR(255)    NOT NULL,
    on_hand_quantity    INTEGER         NOT NULL DEFAULT 0,
    version             BIGINT          NOT NULL DEFAULT 0,
    created_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    CONSTRAINT pk_inventory_stock PRIMARY KEY (product_id),
    CONSTRAINT inventory_stock_on_hand_non_negative CHECK (on_hand_quantity >= 0)
);
CREATE INDEX idx_inventory_stock_product_id ON inventory_service.inventory_stock (product_id);
```

## Production Adapter Design

`JpaInventoryAvailabilityQueryAdapter` issues exactly two queries per `findAvailability` call:

**Query 1 — Stock rows (Spring Data):**
```java
List<InventoryStockEntity> stockEntities = stockRepository.findAllById(productIds);
```
Uses `JpaRepository.findAllById` — a single `IN` query, no N+1.

**Query 2 — Reserved totals (native SQL via EntityManager):**
```sql
SELECT srl.sku_code, COALESCE(SUM(sa.quantity), 0)
FROM inventory_service.stock_reservation_lines srl
JOIN inventory_service.stock_reservations sr ON sr.id = srl.reservation_id
JOIN inventory_service.stock_allocations sa ON sa.line_id = srl.id
WHERE sr.status IN ('RESERVED', 'PARTIALLY_RESERVED')
  AND srl.sku_code IN :skuCodes
GROUP BY srl.sku_code
```
Returns only SKUs with active allocations. Absent rows have zero reserved.

**Merge (Java):**
```java
int available = Math.max(0, onHand - reserved);
```

The adapter is `@Transactional(readOnly = true)`. No pessimistic locks. No mutations.

---

## Query Strategy

Two bounded queries regardless of the number of product IDs. No N+1 behavior. Readability
was preferred over a single complex JOIN-and-aggregate query, following the task's guidance.

## Spring Bean Wiring

`inventoryAvailabilityQueryPort` is registered in `InventoryPersistenceConfiguration` with
`@ConditionalOnMissingBean(InventoryAvailabilityQueryPort.class)`.

The class-level `@ConditionalOnMissingBean(StockReservationRepository.class)` on
`InventoryPersistenceConfiguration` ensures that:

- **Production runtime:** no mocks present → full JPA config loads → production adapter active
- **Non-Testcontainers integration tests:** `InventoryServiceTestConfiguration` provides `StockReservationRepository` and `InventoryAvailabilityQueryPort` → JPA config class skipped
- **Testcontainers adapter tests:** no mock ports provided → JPA config loads → both JPA adapters active
- **gRPC persistence tests:** same as Testcontainers adapter tests; `InventoryAvailabilityGrpcService` injected via Spring

---

## Exception Mapping Correction

**Before:** `NullPointerException → INVALID_ARGUMENT` (incorrect — NPE is a server programming defect, not a client validation error)

**After:**
```java
if (exception instanceof IllegalArgumentException) → INVALID_ARGUMENT
else                                                → INTERNAL
```

`NullPointerException` now maps to `INTERNAL`, consistent with the gRPC specification.

## Known Limitations

1. **`inventory_stock` is populated manually.** This task introduces the table but no stock
   administration API. Test data is inserted via `JdbcTemplate` helpers. A stock replenishment
   API is out of scope per the task definition.

2. **Flyway version numbering shared across locations.** The platform and inventory_service
   migrations share a single Flyway version namespace (due to combined `locations`). The V5
   number was chosen to avoid conflicts with platform V2–V4. Future service-specific migrations
   must continue this shared numbering scheme.

3. **`@EntityScan` covers all entities in the same package.** `InventoryPersistenceConfiguration`
   now scans both `StockReservationEntity` and `InventoryStockEntity`. This is correct because
   both live in `adapter.outbound.persistence`.

## Explicit Confirmations

**No reservation occurs.** `JpaInventoryAvailabilityQueryAdapter.findAvailability` is a
`@Transactional(readOnly = true)` method. It issues two `SELECT` queries and returns snapshots.
No `INSERT`, `UPDATE`, or `DELETE` statement is executed. No pessimistic lock is acquired.

**No in-memory adapter in production.** `ConfigurableInventoryAvailabilityQueryAdapter` lives
in `src/test/java`. `InventoryPersistenceConfiguration` provides `JpaInventoryAvailabilityQueryAdapter`
in the production runtime; no test adapter is wired unless `InventoryAvailabilityQueryPort` is
explicitly provided by a `@TestConfiguration`.

**No commit was made.** Changes are uncommitted per task policy.
