# ARB-022.5.4 — Multi-Worker Claim-Based Polling Runtime

## Context

ARB-022.5.1–5.3 delivered the full claim infrastructure:
- `PublishStatus.CLAIMED`, `claimedBy`, `claimedAt` on `OutboxEvent`
- `claimEvent()` and `claimPending()` on `OutboxRepository`
- `SELECT ... FOR UPDATE SKIP LOCKED` + bulk `UPDATE` in `SpringDataOutboxRepository`

The polling runtime (`OutboxPollingService`, `OutboxPollingScheduler`,
`OutboxSchedulingAutoConfiguration`) still called `findPending(limit)` via
`SequentialPendingDispatchService`. Multiple JVM instances polling concurrently
could retrieve the same PENDING events and dispatch them more than once.

This slice wires the claim-based retrieval into the scheduler so concurrent workers
receive disjoint batches at the database level.

---

## Summary

Three code changes and a restructured auto-configuration complete the runtime:

1. **`ClaimBasedBatchDispatchService`** — new application service. Calls
   `claimPending(workerId, now, limit)` atomically, then dispatches the returned
   CLAIMED events sequentially via `DispatchOutboxMessageService`. Each JVM instance
   carries a unique `workerId` recorded in the claim row.

2. **`OutboxPollingService`** — changed delegate from `SequentialPendingDispatchService`
   to `ClaimBasedBatchDispatchService`. Call site: `dispatchClaimed(batchSize)`.
   Overlap prevention (AtomicBoolean) is unchanged — it guards against multiple
   scheduler ticks within the same JVM, while the database claim guards across JVMs.

3. **`OutboxPollingProperties`** — new `worker-id` property. When blank (default)
   the auto-configuration generates `<hostname>-<8-char UUID suffix>`.

4. **`OutboxSchedulingAutoConfiguration`** — restructured to own the full pipeline.
   Now conditions on `OutboxRepository` + `OutboundMessagePublisher` beans (the two
   service-provided contracts) and auto-creates:
   `DispatchOutboxMessageService` → `ClaimBasedBatchDispatchService` →
   `OutboxPollingService` → `OutboxPollingScheduler`.
   Previously conditioned on `SequentialPendingDispatchService`, which services had
   to create themselves — that responsibility now lives in the auto-configuration.

## Architectural Decisions

### ClaimBasedBatchDispatchService over modifying SequentialPendingDispatchService

`SequentialPendingDispatchService` is a tested, standalone class. Adding `workerId`
and `TimeProvider` to its constructor would force all its callers to change. Creating
a separate class leaves the existing class intact as a tested unit and makes the claim
vs. non-claim distinction explicit.

### Auto-configuration owns the full pipeline

Previously, services had to create `SequentialPendingDispatchService` (and implicitly
`DispatchOutboxMessageService`) as Spring beans themselves, and the auto-configuration
only added the polling wrapper. This created hidden coupling: the auto-configuration
would silently do nothing if a service forgot to create the prerequisite bean.

The restructured auto-configuration conditions on `OutboxRepository` +
`OutboundMessagePublisher` — the two genuinely service-specific contracts — and
creates the entire dispatch-through-scheduling chain internally. Services get the full
pipeline for free when those two beans are present.

### Worker ID generation

The worker ID default (`hostname-XXXXXXXX`) gives operators a human-readable identity
in claim rows and log lines, while the random suffix guarantees uniqueness across
instances on the same host (e.g. multiple pods on a node). The `InetAddress` lookup
falls back to `"unknown"` if the hostname is unavailable rather than failing the
context load.

---

## Implementation Notes

- `SequentialPendingDispatchService` is still present and tested. It is no longer
  wired by the auto-configuration but remains available for services that want manual
  control of the dispatch chain.

- `DispatchOutboxMessageService.dispatch()` does not check the incoming event's
  `publishStatus`. It calls `publisher.publish(event)` and then `markPublished()` or
  `markFailed()`. The JPA adapter's `markPublished` and `markFailed` clear claim
  metadata (`claimedBy`, `claimedAt`) unconditionally — so the transition
  `CLAIMED → PUBLISHED` and `CLAIMED → FAILED` works correctly without changes to
  the dispatch service.

- Stale CLAIMED events (claimed but never published due to worker crash) are not yet
  recovered. That is ARB-022.6 scope.

## Tradeoffs

| Decision | Alternative | Reason chosen |
|---|---|---|
| Separate `ClaimBasedBatchDispatchService` class | Modify `SequentialPendingDispatchService` | Preserves existing tested behavior; makes claim vs. non-claim paths explicit |
| Auto-configuration owns pipeline creation | Services create `DispatchOutboxMessageService` | Reduces service boilerplate; single activation point |
| Hostname + random suffix for worker ID | Pure UUID | Human-readable in operations; still unique per startup |
