## Context

ARB-023.3 completes the gRPC call path from Order Service to Inventory Service for the
availability check that precedes order creation. The slice delivers the outbound port,
client adapter, mapper chain, exception hierarchy, Spring configuration, and full test
coverage including a cross-service PostgreSQL-backed integration test.

Dependent tasks already completed: ARB-023.1 (gRPC contract), ARB-023.2 (Inventory gRPC
server adapter), ARB-023.2A (inventory persistence).

---

## Summary

The Order Service can now query Inventory availability through the production gRPC path.
The adapter maps an Order-owned application model to the generated protobuf contract and
back, keeping all protobuf types within the `adapter.outbound.grpc.inventory` package.
Three cross-service scenarios (AVAILABLE, INSUFFICIENT_STOCK, PRODUCT_NOT_FOUND) are
exercised end-to-end against a Flyway-migrated PostgreSQL container.

## Architectural Decisions

### Port ownership

`InventoryAvailabilityPort` is owned by Order Service (`application.port.outbound`). It
expresses what Order needs from Inventory using Order-owned types (`AvailabilityLineQuery`,
`AvailabilityLineResponse`) — not Inventory types and not protobuf.

### Request/response model

The port uses two simple records that existed before this task:
- `AvailabilityLineQuery(String sku, int requestedQuantity)`
- `AvailabilityLineResponse(String sku, int availableQuantity)`

No new domain value objects were introduced; the existing records fit the semantics.

### Blocking stub

`InventoryAvailabilityServiceGrpc.InventoryAvailabilityServiceBlockingStub` is used. The
application is not reactive, virtual threads handle blocking I/O, and the RPC is unary.
The stub is created once via `GrpcChannelFactory.createChannel("inventory")` and reused
per-request with `.withDeadlineAfter(deadline.toMillis(), MILLISECONDS)` applied each call.

### Spring gRPC client dependency

`net.devh:grpc-client-spring-boot-starter:3.1.0.RELEASE` — symmetric with the inventory
server starter already on the classpath. Channel lifecycle is managed by the starter.

### Configuration gating — `@ConditionalOnProperty`

`OrderInventoryGrpcConfiguration` is annotated `@ConditionalOnProperty(name = "grpc.client.inventory.address")`. The address is NOT set in `application.yml` by default (only via env var in production). This follows the Kafka pattern from CLAUDE.md: tests that do not configure the property skip the config class entirely; test stubs fill the port instead.

A class-level `@ConditionalOnMissingBean(InventoryAvailabilityPort.class)` was tried first
and caused `BeanDefinitionOverrideException` in Spring Boot 4.x Testcontainer tests because
the main condition evaluates before test `@TestConfiguration` beans are registered.
`@ConditionalOnProperty` avoids the race.

### Deadline

Configured via `arbitrier.order.inventory.grpc.deadline-ms` (default 2000 ms). Applied per
invocation via `stub.withDeadlineAfter(...)`.

### Exception hierarchy

Five classes: base `InventoryAvailabilityException` plus four specializations. All four
gRPC statuses required by the acceptance criteria are mapped explicitly. Protocol violations
from the response mapper propagate without re-wrapping. Original gRPC cause is preserved.

### Response mapping

- AVAILABLE → all lines return `requestedQuantity`; any non-empty `unavailableItems` → `InventoryAvailabilityProtocolException`
- UNAVAILABLE → build index from `unavailableItems` by productId; items absent from the index (were available) return `requestedQuantity`; items present return server's `availableQuantity`
- UNSPECIFIED → `InventoryAvailabilityProtocolException`

### Protobuf isolation

No protobuf type crosses into `application` or `domain` packages. ArchUnit rule
`application_must_not_depend_on_grpc_or_protobuf` enforces this.

## Implementation Notes

### Cross-service integration test design

`OrderToInventoryGrpcIntegrationIT` loads `InventoryServiceApplication.class` with an
explicit `classes` array because the order-service module would otherwise auto-discover
`OrderServiceApplication`. The inner `@TestConfiguration` is NOT picked up automatically
when `classes` is explicitly set — it must be included in the array:

```java
@SpringBootTest(
    classes = {InventoryServiceApplication.class, OrderToInventoryGrpcIntegrationIT.Config.class},
    ...
)
```

The `@TestConfiguration` provides only `WarehouseAllocationPort` (a no-op lambda) because
`InventoryServiceConfiguration.reserveStockUseCase` requires it. No other stubs are
needed: `InventoryPersistenceConfiguration` activates (no `StockReservationRepository`
stub is provided) and wires the JPA adapters.

An in-process gRPC server/channel pair is created per test using the Spring-managed
`InventoryAvailabilityGrpcService` bean. `GrpcInventoryAvailabilityAdapter` is then
instantiated with a blocking stub pointing at the in-process channel.

### Failure-path coverage

`DEADLINE_EXCEEDED`, `UNAVAILABLE`, `INVALID_ARGUMENT`, and `INTERNAL` are exercised
in `GrpcInventoryAvailabilityAdapterTest` using a Mockito-mocked blocking stub. These
unit tests verify the full exception mapping chain including cause preservation.

`StatusRuntimeException` never escapes the adapter — it is always mapped to a typed
`InventoryAvailabilityException` subtype.

## Tradeoffs

- **No retry in this slice** — per task scope; the configured deadline fails clearly.
- **No gRPC metadata propagation** — `traceparent` and `X-Correlation-Id` propagation via
  gRPC metadata is not yet implemented. The business `requestId` field is populated per call.
  This is an explicit gap to address in a future slice.
- **Blocking stub** — chosen over async for simplicity and compatibility with the current
  non-reactive application model.
- **Cross-service IT requires inventory-service JAR** — `inventory-service` is added as a
  `test` scope dependency in order-service pom.xml. This is an intentional cross-module
  coupling in test code only; the production boundary remains clean.

## Open Questions

- **OPEN QUESTION:** gRPC metadata propagation for `traceparent` and `X-Correlation-Id` —
  how should the interceptor be registered and what headers map to which gRPC metadata keys?
- **OPEN QUESTION:** Should `InventoryAvailabilityPort` return a sealed `Available/Unavailable`
  result type rather than `List<AvailabilityLineResponse>`? The current flat list is simpler
  but doesn't distinguish AVAILABLE from UNAVAILABLE at the port contract level. The
  distinction is currently left to the caller to infer from the quantities.
