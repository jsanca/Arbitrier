## Scenarios

### Scenario 1 — Success with duplicate SKUs (`duplicate_skus_normalized_order_and_outbox_persisted`)

Input: SKU-A qty 3, SKU-B qty 1, SKU-A qty 2. Inventory stub returns `Available`.

Verified:
- HTTP 201 with non-blank `orderId` and `status = PENDING`
- Exactly one row in `order_service.orders` with `status = PENDING`
- HTTP `orderId` matches the persisted row's `id`
- `order_service.order_lines` has exactly 2 rows: SKU-A qty 5 (3+2), SKU-B qty 1
- Exactly one row in `order_service.outbox_events` with `event_type = OrderCreatedDomainEvent`
- Outbox `aggregate_id` matches `orderId`
- Outbox payload JSON `lines` has exactly 2 entries

### Scenario 2 — Business rejection via real gRPC (`insufficient_inventory_via_grpc_returns_422_and_nothing_persisted`)

Inventory stub returns `Unavailable` for all items. This exercises the full gRPC path:
`InventoryAvailabilityGrpcService` → `AVAILABILITY_STATUS_UNAVAILABLE` protobuf response
→ `GrpcInventoryAvailabilityAdapter` maps to `AvailabilityLineResponse` with `availableQuantity = 1`
→ `SubmitCorporateBulkOrderService.verifyInventoryAvailability()` detects insufficiency.

Verified: HTTP 422 `ORDER_ITEMS_UNAVAILABLE`, zero orders, zero outbox events.

### Scenario 3 — Protocol failure via real gRPC exception path (`grpc_invalid_argument_maps_to_502_and_nothing_persisted`)

Stub use case throws `IllegalArgumentException`. Production exception chain:

1. `InventoryAvailabilityGrpcService` catches it → `InventoryGrpcExceptionMapper.toStatusException()` → gRPC `INVALID_ARGUMENT`
2. `GrpcInventoryAvailabilityAdapter` catches `StatusRuntimeException(INVALID_ARGUMENT)` → `InventoryAvailabilityGrpcExceptionMapper.toAdapterException()` → `InventoryAvailabilityProtocolException`
3. `OrderInventoryExceptionHandler.handleInventoryProtocolError()` → HTTP 502 `INVENTORY_PROTOCOL_ERROR`

All production exception mapping code runs unmodified. The seam is that `INVALID_ARGUMENT` originates from the stub rather than from a malformed protobuf frame (which the real production response mapper cannot produce).

Verified: HTTP 502 `INVENTORY_PROTOCOL_ERROR`, zero orders, zero outbox events.

### Scenario 4 — Authorization denial (`unauthorized_customer_returns_403_and_no_order_saved`)

`@MockitoBean CustomerAccessPort` configured to deny. Proves authorization is checked before inventory is called.

Verified: HTTP 403 `ORDER_ACCESS_DENIED`, zero orders, zero outbox events.

### Scenarios 5–6 — Bean Validation (`zero_quantity_returns_400`, `negative_quantity_returns_400`)

Bean Validation on `CreateOrderLineRequest.quantity` (`@Min(1)`) rejects quantity 0 and negative. Verified: HTTP 400, zero orders, zero outbox events.

## Test Infrastructure Design

`CorporateBulkOrderVerticalIT.Config` provides an `InventoryAvailabilityPort` bean backed by a real in-process gRPC server. The `AtomicReference<CheckInventoryAvailabilityUseCase>` (`Config.STUB`) is reset in `@BeforeEach` to `Available` and overridden per test for non-happy-path scenarios. The gRPC server and channel are closed in `@AfterAll`.

This design means all three vertical scenarios run in the same Spring context (single Testcontainers startup), making the test class fast and avoiding repeated container overhead.
