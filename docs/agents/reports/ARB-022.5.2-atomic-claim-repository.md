# ARB-022.5.2 — Atomic Claim Repository

## Overview

This slice introduces the repository operation that atomically transitions a
`PENDING` outbox event to `CLAIMED`, recording the worker identity and claim
timestamp. Only one worker can claim a given event: the database enforces
mutual exclusion via a conditional `UPDATE`.

---

## Repository API

`OutboxRepository` gains one new operation (already declared in ARB-022.5.1):

```java
Optional<OutboxEvent> claimEvent(UUID eventId, String workerId, Instant claimedAt);
```

| Return | Meaning |
|--------|---------|
| `Optional.of(event)` | Claim succeeded; returned event is in `CLAIMED` state |
| `Optional.empty()` | Event not found, or not in `PENDING` status |

No exception is thrown for a failed claim — the loser receives `empty()`.

---

## SQL Strategy

The claim is a single conditional `UPDATE` executed via JPQL in
`SpringDataOutboxRepository`:

```java
@Modifying(flushAutomatically = true, clearAutomatically = true)
@Query("""
        UPDATE OutboxEventEntity e
        SET e.publishStatus = 'CLAIMED',
            e.claimedBy     = :workerId,
            e.claimedAt     = :claimedAt
        WHERE e.id = :eventId
          AND e.publishStatus = 'PENDING'
        """)
int claimPending(@Param("eventId") UUID eventId,
                 @Param("workerId") String workerId,
                 @Param("claimedAt") Instant claimedAt);
```

`flushAutomatically = true` — the EntityManager cache is flushed to the
database before the bulk update, preventing stale reads from the L1 cache
from masking a concurrent claim.

`clearAutomatically = true` — the EntityManager cache is cleared after the
update, so the subsequent `findById` read reflects the committed row.

`WHERE e.publishStatus = 'PENDING'` — the database enforces the guard.
If two transactions execute the `UPDATE` concurrently, only one will match a
`PENDING` row; the other will see zero rows updated and the adapter returns
`empty()`.

---

## Adapter Implementation

`JpaOutboxRepositoryAdapter.claimEvent()`:

```java
@Override
public Optional<OutboxEvent> claimEvent(
        final UUID eventId, final String workerId, final Instant claimedAt) {
    Require.notNull(eventId, "eventId");
    Require.notBlank(workerId, "workerId");
    Require.notNull(claimedAt, "claimedAt");
    int affected = repository.claimPending(eventId, workerId, claimedAt);
    if (affected == 0) {
        return Optional.empty();
    }
    return repository.findById(eventId).map(JpaOutboxRepositoryAdapter::toRecord);
}
```

The adapter validates inputs, delegates to the atomic update, and reads
back the updated row only when the claim succeeded. No polling, retry, or
scheduler logic is introduced.

---

## In-Memory Implementation

`InMemoryOutboxRepository.claimEvent()` is `synchronized` to protect the
in-memory list from concurrent test threads:

```java
@Override
public synchronized Optional<OutboxEvent> claimEvent(
        final UUID eventId, final String workerId, final Instant claimedAt) {
    ...
    for (int i = 0; i < events.size(); i++) {
        OutboxEvent e = events.get(i);
        if (e.eventId().equals(eventId)) {
            if (e.publishStatus() != PublishStatus.PENDING) {
                return Optional.empty();
            }
            OutboxEvent claimed = e.claim(workerId, claimedAt);
            events.set(i, claimed);
            return Optional.of(claimed);
        }
    }
    return Optional.empty();
}
```

## Concurrency Guarantees

The database is the authority. Two workers racing on the same `PENDING` row:

1. Worker A executes `UPDATE ... WHERE status='PENDING'` → 1 row updated → proceeds
2. Worker B executes the same `UPDATE` → 0 rows updated (status is now `CLAIMED`)
   → receives `Optional.empty()`

No exception is thrown for the loser. No application-level lock is used.
PostgreSQL row-level locking ensures exactly one winner per row per transaction.

---

## Limitations

- **No batch claim**: this slice acquires one event at a time. Batch claim
  (`claimNextBatch`) is out of scope; ARB-022.5.3 or later will address it.
- **No lease expiration**: a claimed event that is never published or failed
  will remain in `CLAIMED` status indefinitely until explicit recovery logic
  is introduced (future slice).
- **No retry on failed claim**: the caller receives `empty()` and decides
  what to do. The dispatching layer owns retry.
- **In-memory atomicity**: `synchronized` on `InMemoryOutboxRepository`
  serializes all claim attempts; it does not replicate database transaction
  isolation semantics (e.g., no MVCC, no serialization failure).

## Architecture Verification

- No polling logic in the repository adapter — `claimEvent` is a pure persistence
  operation.
- No scheduler dependency introduced.
- No retry behavior inside the repository.
- No worker orchestration in this slice.
- `InMemoryOutboxRepository` implements the same port contract and is available
  for test substitution in all service integration tests.
