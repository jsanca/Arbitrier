# ARB-023.3.3 — Inventory Availability Response Contract: Implementation Report

## Context

ARB-023.3.1 identified a correctness defect: `SubmitCorporateBulkOrderService.verifyInventoryAvailability()` uses `Stream.allMatch()` on the responses list. `allMatch` on an empty stream returns `true` vacuously — meaning an inventory port implementation that returns fewer responses than queries (or an empty list) would silently approve the order.

The architecture review decided that malformed remote responses are upstream protocol failures, not business unavailability, and must be caught at the outbound adapter boundary before the application service sees them.

---

## Summary

- `GrpcInventoryAvailabilityAdapter` now validates the mapped response against the original query list before returning. Any mismatch (count, duplicate SKU, unexpected SKU) throws `InventoryAvailabilityProtocolException`.
- `OrderInventoryExceptionHandler` was split: `InventoryAvailabilityProtocolException` maps to 502 Bad Gateway (`INVENTORY_PROTOCOL_ERROR`); `InventoryAvailabilityInternalException` remains 500 (`INVENTORY_INTEGRATION_ERROR`).
- 7 new adapter unit tests cover all required scenarios. The cross-service gRPC integration test (`OrderToInventoryGrpcIntegrationIT`) also exercises the new validation on the production code path.
- A pre-existing validation gap in `CreateOrderRestMapper.toResponse()` was discovered and fixed — the Javadoc claimed it threw on null/blank but the implementation did not.
- Full suite: **BUILD SUCCESS**.

## Responsibility Boundary

| Concern | Owner |
|---|---|
| Did Inventory answer every requested SKU? | `GrpcInventoryAvailabilityAdapter.validateResponseContract()` |
| Are there duplicate or unexpected SKUs? | `GrpcInventoryAvailabilityAdapter.validateResponseContract()` |
| Is the protobuf status field valid? | `InventoryAvailabilityGrpcResponseMapper.toLines()` (existing — UNSPECIFIED → protocol exception) |
| Is the returned quantity sufficient? | `SubmitCorporateBulkOrderService.verifyInventoryAvailability()` |

The application service trusts the response it receives from the port. The adapter guarantees the response is structurally valid before returning it.

## Validation Method

```java
private void validateResponseContract(
        final List<AvailabilityLineResponse> result,
        final List<AvailabilityLineQuery> queries) {

    if (result.size() != queries.size()) {
        throw new InventoryAvailabilityProtocolException(
                "Inventory response line count mismatch: expected " + queries.size()
                + ", got " + result.size());
    }

    final Set<String> requestedSkus = queries.stream()
            .map(AvailabilityLineQuery::sku).collect(Collectors.toSet());

    final Set<String> seenSkus = new HashSet<>(result.size());
    for (final AvailabilityLineResponse r : result) {
        if (!seenSkus.add(r.sku())) {
            throw new InventoryAvailabilityProtocolException(
                    "Inventory response contains duplicate SKU: " + r.sku());
        }
        if (!requestedSkus.contains(r.sku())) {
            throw new InventoryAvailabilityProtocolException(
                    "Inventory response contains unexpected SKU: " + r.sku());
        }
    }
}
```

## HTTP Mapping — Updated

| Exception | HTTP | Code |
|---|---|---|
| `InventoryAvailabilityTimeoutException` | 504 Gateway Timeout | `INVENTORY_TIMEOUT` |
| `InventoryAvailabilityRemoteUnavailableException` | 503 Service Unavailable | `INVENTORY_SERVICE_UNAVAILABLE` |
| `InventoryAvailabilityProtocolException` | **502 Bad Gateway** | **`INVENTORY_PROTOCOL_ERROR`** |
| `InventoryAvailabilityInternalException` | 500 Internal Server Error | `INVENTORY_INTEGRATION_ERROR` |

Previously both protocol and internal failures mapped to 500 `INVENTORY_INTEGRATION_ERROR`.

## Tests Added

### `GrpcInventoryAvailabilityAdapterTest` (7 new)

The `adapterWithMappedResult()` test helper injects a mock response mapper via the package-private constructor, allowing the validator to be exercised with data that the real gRPC protocol cannot produce.

| Test | Scenario |
|---|---|
| `complete_response_is_accepted` | Full, correct response passes |
| `empty_response_for_nonempty_request_throws_protocol_exception` | Empty list for 1+ queries → protocol exception |
| `missing_requested_sku_throws_protocol_exception` | Fewer results than queries → protocol exception |
| `duplicate_response_sku_throws_protocol_exception` | Same SKU twice → protocol exception |
| `unexpected_response_sku_throws_protocol_exception` | Response contains SKU not in request → protocol exception |
| `response_in_different_order_from_request_is_accepted` | Different response order is valid |
| `insufficient_quantity_is_returned_normally_and_not_treated_as_protocol_error` | Quantity below request is a business response, not a protocol error |

### `SubmitCorporateBulkOrderControllerTest` (1 updated)

`inventory_protocol_error_returns_502` — asserts `InventoryAvailabilityProtocolException` → 502 + `INVENTORY_PROTOCOL_ERROR`.

### `CreateOrderRestMapperTest` (4 pre-existing tests now fixed)

The mapper gap was discovered during the test run — 4 tests for null/blank validation in `toResponse()` were already in the test file but had no corresponding implementation. Fixed by adding `Require.notBlank` guards.
