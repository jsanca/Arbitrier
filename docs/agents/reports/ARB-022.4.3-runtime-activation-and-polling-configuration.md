# ARB-022.4.3 — Runtime Activation & Polling Configuration

## Configuration Model

**Class:** `OutboxPollingProperties`  
**Package:** `com.arbitrier.platform.messaging.outbox.spring`  
**Prefix:** `arbitrier.messaging.outbox.polling`

| Property | Default | Meaning |
|----------|---------|---------|
| `enabled` | `true` | Activates the polling scheduler on startup |
| `initial-delay-ms` | `5 000` | Milliseconds before the first polling cycle fires |
| `fixed-delay-ms` | `10 000` | Milliseconds between end of one cycle and start of the next |
| `batch-size` | `100` | Maximum messages retrieved per polling cycle |

`OutboxPollingProperties` is a `@ConfigurationProperties` bean, not a set of scattered `@Value` fields. All polling tuning is readable and documented in one place.

## Activation Policy

`OutboxSchedulingAutoConfiguration` activates when:

1. `arbitrier.messaging.outbox.polling.enabled=true` (default; omitting the property is treated as enabled).
2. A `SequentialPendingDispatchService` bean is present — confirming the full dispatch chain (repository, dispatcher, publisher) has been wired by the application.

When `enabled=false`, neither `OutboxPollingService` nor `OutboxPollingScheduler` is registered. No no-op beans are created.

## What Changed in This Slice

- `OutboxSchedulingAutoConfiguration` now creates `OutboxPollingService` with `batchSize` from `OutboxPollingProperties`. Previously the configuration was `@ConditionalOnBean(OutboxPollingService.class)`, expecting someone else to create it.
- The condition changed from `@ConditionalOnBean(OutboxPollingService.class)` to `@ConditionalOnBean(SequentialPendingDispatchService.class)`, which is the actual lower-layer dependency.
- `@ConditionalOnProperty` for `enabled` was added.
- `@EnableConfigurationProperties(OutboxPollingProperties.class)` was added.

The `OutboxPollingScheduler` `@Scheduled` annotation retains its property placeholders (`fixed-delay-ms`, `initial-delay-ms`) with inline fallbacks matching the `OutboxPollingProperties` defaults. `@Scheduled` reads directly from the Spring Environment; the `@ConfigurationProperties` bean is the authoritative documentation and provides `enabled` and `batchSize` which `@Scheduled` cannot express.

## Property Namespace

All properties live under `arbitrier.messaging.outbox.polling`:

```yaml
arbitrier:
  messaging:
    outbox:
      polling:
        enabled: true
        initial-delay-ms: 5000
        fixed-delay-ms: 10000
        batch-size: 100
```

### `OutboxSchedulingAutoConfigurationTest` (`ApplicationContextRunner` — no Kafka, no DB)

| Test | Scenario |
|------|----------|
| `scheduler_and_polling_service_registered_when_enabled_by_default` | Default → both beans present |
| `scheduler_registered_when_enabled_explicitly` | `enabled=true` → scheduler present |
| `scheduler_absent_when_disabled` | `enabled=false` → no beans |
| `scheduler_absent_when_no_sequential_dispatch_service` | No dispatch service → no beans |

| `default_batch_size_is_propagated_to_polling_service` | Default → `dispatchPending(100)` called |
