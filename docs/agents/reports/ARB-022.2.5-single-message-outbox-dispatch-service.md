# ARB-022.2.5 — Single-Message Outbox Dispatch Service

## Service Name and Package

**Class:** `DispatchOutboxMessageService`  
**Package:** `com.arbitrier.platform.messaging.outbox.application`

The package is new. It sits below the outbox contract package and holds coordination logic that depends on the outbox ports without being a port itself.

The service depends only on:
- `OutboundMessagePublisher` (port)
- `OutboxRepository` (port)
- `OutboxEvent` and `Require` (platform types)

No Kafka imports. No framework dependency.

## Public API

```java
public CompletionStage<Void> dispatch(OutboxEvent message)
```

Null input is rejected immediately via `Require.notNull`. All other outcomes (async success, async failure, immediate publication failure) follow the flows below.

## Success Flow

```
publisher.publish(event)
  ↓ broker ack (CompletionStage completes normally)
outboxRepository.markPublished(event.eventId())
  ↓ (short DB transaction, completes normally)
returned stage completes normally
```

If `markPublished` throws, the returned stage completes exceptionally (via `thenRun` catching the thrown exception as a `CompletionException`). `markFailed` is NOT called on a `markPublished` failure — treated as an unresolved operational error for future retry/recovery handling.

## Failure Flow

### Asynchronous failure

```
publisher.publish(event)
  ↓ transport failure (CompletionStage completes exceptionally with pubEx)
outboxRepository.markFailed(event.eventId())
  ↓ (if markFailed also throws, exception added as suppressed to pubEx)
returned stage completes exceptionally with pubEx
```

### Immediate failure (publish throws before returning a stage)

```
publisher.publish(event) throws immediately
  → markFailed(event.eventId()) called
  → original exception rethrown (markFailed exception added as suppressed if needed)
```

## Composition Strategy

The async flows use `CompletionStage` composition without blocking:

```java
return published
    .exceptionallyCompose(pubEx -> {
        callMarkFailed(message, pubEx);        // marks failed, attaches suppressed
        return CompletableFuture.failedFuture(pubEx);
    })
    .thenRun(() -> outboxRepository.markPublished(message.eventId()));
```

- `exceptionallyCompose` fires only on publication failure. On success it passes through to `thenRun`.
- `thenRun(markPublished)` fires only after successful `exceptionallyCompose` pass-through — i.e., only when publication succeeded.
- Neither `get()` nor `join()` is called anywhere. The service is fully non-blocking.

## Transaction Decision

This service is intentionally not `@Transactional`. Holding a database connection open while waiting for Kafka acknowledgement would exhaust the connection pool under load. `markPublished` and `markFailed` each run in their own short, independent transactions managed by the `OutboxRepository` implementation.

## Error-Preservation Behavior

- The original publication exception (`pubEx`) is always the primary cause.
- Any `markFailed` exception is added as a suppressed exception on `pubEx` via `Throwable.addSuppressed()`.
- The publication exception is never silently swallowed.
- Immediate failures rethrow the original exception directly; the returned stage may be absent.

## Deliberately Deferred Concerns

| Concern | Reason |
|---------|--------|
| `findPending()` integration | Requires polling or scheduling — separate slice |
| Claim / in-progress status | Requires schema change and locking strategy |
| Retry counters / backoff | Requires retry policy decisions |
| DLQ routing | Requires DLQ configuration |
| Metrics / tracing | Cross-cutting; separate instrumentation slice |
| Distributed locking | Separate operational concern |

## Tests

| Test | Scenario |
|------|----------|
| `rejects_null_message` | Null guard |
| `calls_publisher_exactly_once` | Publisher invocation count |
| `successful_publication_calls_markPublished` | Happy path — markPublished called |
| `successful_publication_does_not_call_markFailed` | Happy path — markFailed never called |
| `asynchronous_publication_failure_calls_markFailed` | Async failure path |
| `asynchronous_publication_failure_remains_exceptional` | Returned stage is exceptionally completed |
| `immediate_publication_failure_calls_markFailed` | Synchronous throw from publish |
| `markFailed_failure_is_attached_as_suppressed_to_publication_failure` | Suppressed exception attachment |
| `markPublished_failure_completes_exceptionally` | markPublished exception propagates; markFailed not called |
