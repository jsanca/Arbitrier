# ARB-022.2.1 — Messaging Publisher Port

## Context

ARB-022.1 established the outbound message model (`OutboxEvent`, `MessageNature`, `OutboundRoutingStrategy`). This slice introduces the runtime-neutral port that an outbox drainer will use to hand a persisted message to a transport adapter.

---

## Interface Name Rationale

Chosen: **`OutboundMessagePublisher`**

| Candidate | Reason rejected |
|-----------|----------------|
| `OutboxPublisher` | Ties the name to the outbox storage mechanism; callers should not need to know how messages are persisted |
| `KafkaPublisher` | Transport-specific — violates transport independence |
| `MessagePublisher` | Acceptable but ambiguous at the platform level; "outbound" makes the direction explicit |
| `send(...)` method name | Implies point-to-point delivery; echoes low-level transport APIs (`KafkaTemplate.send()`) |
| **`publish(...)` method name** | **Chosen** — expresses intent without implying delivery mode; handles both events and commands through a single call |

---

## Package Location

`com.arbitrier.platform.messaging.outbox`

Rationale: the port operates on `OutboxEvent` — a type already in this package. Keeping publisher and event type co-located avoids cross-package coupling in the transport-neutral layer.

---

## Public API

```java
public interface OutboundMessagePublisher {
    void publish(OutboxEvent message);
}
```

**`void` return type:** the outbox pattern already tracks delivery outcomes via `OutboxRepository.markPublished()` / `markFailed()`. The publisher's responsibility is dispatch only; it does not need to return a result to the drainer.

**Unchecked exceptions:** transport errors propagate as runtime exceptions; the drainer is responsible for catching and calling `markFailed`.

**Single method:** makes the interface a functional interface, enabling lambda implementation in tests and trivial test doubles without a mocking framework.

---

## Pre-existing Stub Handling

`OutboxPublisher.java` existed as an empty placeholder (no callers, no API). It is updated to `@Deprecated` and extended from `OutboundMessagePublisher` for binary compatibility. Callers of `OutboxPublisher` (none currently) will automatically satisfy `OutboundMessagePublisher`.

---

## Architectural Intent

The outbox drainer (a future slice) will:
1. Call `OutboxRepository.findPending()` to load unsent messages.
2. Pass each `OutboxEvent` to `OutboundMessagePublisher.publish(message)`.
3. On success: call `OutboxRepository.markPublished(eventId)`.
4. On failure: call `OutboxRepository.markFailed(eventId)`.

The Kafka adapter will implement `OutboundMessagePublisher` using `KafkaTemplate` and `OutboundRoutingStrategy` to resolve the topic. No Kafka types appear in the port or in application services.
