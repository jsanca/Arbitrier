# ARB-022.3.1 — Pending Message Retrieval

## Repository API

A new method was added to `OutboxRepository` alongside the existing parameterless `findPending()`:

```java
List<OutboxEvent> findPending(int limit);
```

| Input | Behaviour |
|-------|-----------|
| `limit < 0` | Throws `IllegalArgumentException` immediately |
| `limit == 0` | Returns empty list (no store access) |
| `limit > 0` | Returns at most `limit` pending events, oldest first |

The existing `findPending()` (no limit) is unchanged; it continues to be used by the existing JPA adapter tests and the in-memory test adapter.

## Ordering Decision

**Field:** `OutboxEvent#occurredAt` ascending.

`occurredAt` is the closest equivalent to "created at" in the outbox schema — it is set at the time the aggregate writes the event to the outbox and is never modified afterward. Ordering by `occurredAt` ascending gives the dispatch runtime a FIFO delivery order: the oldest pending event is always dispatched first.

No additional ordering column was introduced. If two events share the same `occurredAt` (e.g. events created in rapid succession within the same millisecond), their relative order is stable within a single query execution but is not guaranteed to be the same across separate queries.

## Pending Definition

Only events in `PublishStatus.PENDING` are returned. Events in `PUBLISHED` or `FAILED` state are excluded. No new status values (`CLAIMED`, `IN_PROGRESS`, `RETRYING`) were introduced — those belong to future runtime slices.

## Limit Semantics

The repository never returns more than `limit` events. The JPA adapter uses `PageRequest.of(0, limit)` with Spring Data's `Pageable` support — this applies `LIMIT` at the SQL level without a count query. The in-memory adapter uses `Stream.limit(limit)`. Both implementations skip the store entirely when `limit == 0`.

## Persistence Implementation

**`SpringDataOutboxRepository`** — new derived query method:
```java
List<OutboxEventEntity> findByPublishStatusOrderByOccurredAtAsc(String publishStatus, Pageable pageable);
```
Spring Data JPA derives the `WHERE publish_status = ?` predicate, the `ORDER BY occurred_at ASC` clause, and the `LIMIT` from the method name and `Pageable` argument. No custom JPQL or native SQL required.

**`JpaOutboxRepositoryAdapter.findPending(int limit)`:**
```java
if (limit < 0) throw new IllegalArgumentException("limit must not be negative: " + limit);
if (limit == 0) return List.of();
return repository.findByPublishStatusOrderByOccurredAtAsc(
        PublishStatus.PENDING.name(), PageRequest.of(0, limit))
    .stream().map(JpaOutboxRepositoryAdapter::toRecord).toList();
```

**`InMemoryOutboxRepository.findPending(int limit)`** mirrors the JPA semantics using `Stream.sorted(Comparator.comparing(OutboxEvent::occurredAt)).limit(limit)`.

## Tests

| Test class | Prior | Added | Total |
|---|---|---|---|
| `JpaOutboxRepositoryAdapterTest` | 7 | 7 | 14 |
| `InMemoryOutboxRepositoryTest` | 8 | 6 | 14 |

New JPA adapter test scenarios:
- empty repository → empty result
- single pending event returned
- ordering: 3 events saved in non-chronological order → oldest-first in result
- limit larger than available → all returned
- `limit = 0` → empty list
- `limit < 0` → `IllegalArgumentException`
- published and failed events excluded

New in-memory test scenarios mirror the JPA scenarios (no DB needed).
