# ARB-022.4.1 — Polling Cycle Overlap Prevention

## Service

**Class:** `OutboxPollingService`  
**Package:** `com.arbitrier.platform.messaging.outbox.application`

This slice modified `OutboxPollingService` (introduced in ARB-022.3.3) to prevent overlapping polling cycles.

## Overlap Scenario

`OutboxPollingService.pollOnce()` delegates to `SequentialPendingDispatchService.dispatchPending(batchSize)`, which returns a `CompletionStage<Void>`. The stage completes asynchronously. If a future scheduler fires at a fixed interval, a second `pollOnce()` invocation can arrive while the first cycle's stage is still executing — for example, when the batch is large or a message dispatch is slow.

Without overlap prevention, this would result in two concurrent dispatch chains reading from the same pending-message window, potentially dispatching the same messages twice.

## Chosen Synchronization Mechanism

`AtomicBoolean running` — a single field initialized to `false`. `compareAndSet(false, true)` atomically claims the cycle; `running.set(false)` releases it.

No lock, no semaphore, no executor. The scheduler sees a lightweight flag check on each invocation.

## Skip Policy

```
pollOnce()
  if running == false  →  set running = true
                           dispatch cycle
                           clear running when stage completes
                           return cycle stage

  if running == true   →  return completedFuture(null)  [no dispatch]
```

The skip returns an already-completed stage rather than throwing or blocking. The scheduler can treat it like a normal completion — nothing to log, nothing to retry. The in-flight cycle will process any messages that arrived before it finishes.

## Running Flag Clearing — All Paths

The flag is cleared in `whenComplete`, which fires regardless of success or failure:

```java
return cycle.whenComplete((result, ex) -> running.set(false));
```

For a synchronous throw from the delegate, the flag is cleared in a `catch` block before rethrowing:

```java
try {
    cycle = sequentialDispatch.dispatchPending(batchSize);
} catch (RuntimeException immediate) {
    running.set(false);
    throw immediate;
}
```

This ensures no polling cycle can permanently block future invocations.

## Why Scheduler Remains Simple

The scheduler will call `pollOnce()` and receive a `CompletionStage<Void>`. It does not need to check, read, or reset any running state — that is fully encapsulated in `OutboxPollingService`. The scheduler fires on a fixed interval and trusts the polling facade to decide whether a new cycle starts or is skipped.

## Tests

| Test | Scenario |
|------|----------|
| `valid_batch_size_accepted` | Positive batchSize → construction succeeds |
| `zero_batch_size_rejected` | `batchSize=0` → `IllegalArgumentException` |
| `negative_batch_size_rejected` | `batchSize=-1` → `IllegalArgumentException` |
| `first_poll_executes_and_delegates` | No in-flight cycle → `dispatchPending` called once |
| `configured_batch_size_is_passed_to_dispatchPending` | Exact value forwarded |
| `concurrent_poll_returns_immediately_without_dispatch` | Second call during active cycle → done stage, no extra dispatch |
| `dispatcher_invoked_only_once_when_overlap_occurs` | Three concurrent calls → `dispatchPending` called once |
| `running_flag_cleared_after_success_so_next_poll_executes` | After success, next call dispatches again |
| `running_flag_cleared_after_failure_so_next_poll_executes` | After failure, next call dispatches again |
| `running_flag_cleared_after_immediate_exception` | After synchronous throw, next call dispatches again |
| `successful_stage_is_propagated` | Done stage passes through |
| `exceptional_stage_is_propagated` | Failed stage passes through |
| `immediate_delegate_failure_is_propagated` | Synchronous throw propagates as-is |
