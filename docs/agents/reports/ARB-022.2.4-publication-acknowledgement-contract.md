# ARB-022.2.4 — Publication Acknowledgement Contract

## Contract Evolution

`OutboundMessagePublisher.publish()` changed from:

```java
void publish(OutboxEvent message)
```

to:

```java
CompletionStage<Void> publish(OutboxEvent message)
```

The previous `void` return discarded the `CompletableFuture<SendResult<K,V>>` returned by `KafkaTemplate.send()`. An outbox drainer that must call `markPublished()` after confirmed delivery — or `markFailed()` after transport failure — has no signal to act on. This slice introduces that signal.

## Why `CompletionStage<Void>`

| Option | Rejected because |
|--------|-----------------|
| `CompletableFuture<Void>` | Exposes the concrete type; callers can call `complete()`, `cancel()`, etc. |
| `ListenableFuture` | Spring-specific; leaks infrastructure into a platform contract |
| `SendResult<K,V>` | Kafka-specific; violates the transport-neutral boundary |
| Custom result type | No concrete need identified; `CompletionStage` is standard JDK |

`CompletionStage` is:
- Part of the JDK (`java.util.concurrent`) — zero additional dependency
- A read-only view of asynchronous completion
- Transport-neutral — implementations may back it with any future type
- Composable — callers can chain `.thenRun()`, `.exceptionally()` etc. without blocking

## Success Semantics

The returned stage completes normally (`null` value, `Void` type) when Kafka acknowledges the record. The adapter maps `SendResult` away with `thenApply(sendResult -> null)` — `SendResult` never escapes the Kafka adapter class.

## Failure Semantics

- **Immediate failures** (null argument, routing strategy exception, serializer exception) still throw directly before a stage is returned. This is intentional: they indicate programming errors or missing configuration, not transport degradation.
- **Asynchronous Kafka failures** (broker unavailable, send timeout) complete the returned stage exceptionally. The exception propagates through `CompletableFuture`'s standard composition: when the underlying `CompletableFuture<SendResult>` completes exceptionally, `thenApply()` is skipped and the resulting `CompletableFuture<Void>` is also completed exceptionally with the same cause.

No exceptions are swallowed. No `get()` or `join()` calls are introduced.

## Kafka Adapter Change

```java
return kafkaTemplate.send(record).thenApply(sendResult -> null);
```

`KafkaTemplate.send(ProducerRecord)` returns `CompletableFuture<SendResult<String, String>>`. The `.thenApply(sendResult -> null)` call:
1. Discards the `SendResult` — it contains Kafka internals the drainer does not need.
2. Returns `CompletableFuture<Void>` which is a valid `CompletionStage<Void>`.
3. Propagates exceptions from the underlying future automatically.

## `OutboxPublisher` (deprecated)

`OutboxPublisher extends OutboundMessagePublisher` inherits the new `CompletionStage<Void>` contract automatically. No production callers exist; no compatibility shim was introduced.

## Why `SendResult` Remains Internal

`SendResult` carries Kafka partition and offset metadata. The outbox drainer only needs to know "succeeded" or "failed" — it does not need partition or offset. Exposing `SendResult` in the port contract would couple all callers to the Kafka transport and make it impossible to swap transports without modifying call sites.
