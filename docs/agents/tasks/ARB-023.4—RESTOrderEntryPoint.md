## Objective

Complete the production REST entry path for order creation in `order-service`.

The inbound REST adapter must accept an order-creation request, obtain the authenticated user identity from the existing security integration, invoke the order application use case, and return the appropriate HTTP response.

Before creating an order, the application flow must verify product availability through the existing `InventoryAvailabilityPort`.

ARB-023.3 already provides the production gRPC adapter behind that port.

---

## Required Call Path

```text
HTTP POST
    │
    ▼
REST Controller / Inbound Adapter
    │
    ▼
Create Order Input Port
    │
    ├── validate request
    ├── verify customer access
    ├── query InventoryAvailabilityPort
    ├── reject unavailable items
    ├── create Order aggregate
    ├── persist Order
    └── publish/store required creation event
    │
    ▼
HTTP Response
```

Confirm the actual existing application flow before modifying it. Do not duplicate logic already implemented in ARB-007 or later slices.

## Scope

### 1. Inspect the current order creation path

Identify:

* Existing REST controller or inbound adapter.
* Existing create-order input port/use case.
* Existing request and response DTOs.
* Existing `CustomerAccessPort`.
* Existing authentication principal-to-`UserId` mapping.
* Existing order repository and event publication behavior.
* Existing validation and global error-handling conventions.

Prefer extending the existing design over introducing parallel abstractions.

### 2. Wire inventory availability into the application flow

The application layer must call:

```java
InventoryAvailabilityPort
```

using Order-owned types.

Do not expose protobuf or gRPC types outside:

```text
adapter.outbound.grpc.inventory
```

For each requested order line:

* Send the SKU/product identifier.
* Send the requested quantity.
* Compare the returned available quantity against the requested quantity.

The order must not be created when any line has insufficient availability.

Unknown products reported as zero availability must also prevent order creation.

### 3. Preserve responsibility boundaries

The REST adapter must:

* Parse the HTTP request.
* Obtain the authenticated user identity.
* Map the request into an application command.
* Invoke the application input port.
* Map the application result into an HTTP response.

The REST adapter must not:

* Call the gRPC stub directly.
* Call `InventoryAvailabilityPort` directly unless the current architecture explicitly defines the REST adapter as the application orchestrator.
* Contain inventory business rules.
* Construct or persist the `Order` aggregate directly.
* Depend on protobuf classes.

Inventory availability belongs in the application use case.

### 4. Define or reuse the unavailable-inventory outcome

Prefer an explicit application/domain outcome over leaking technical exceptions.

Use the project’s existing result/error model if available.

The failure should identify enough information for the client to understand which line or lines are unavailable, without exposing internal inventory-service details.

Potential semantic outcome:

```text
ORDER_ITEMS_UNAVAILABLE
```

Include line-level details only if consistent with the current API error contract.

Do not use a generic HTTP 500 for business unavailability.

### 5. HTTP behavior

Confirm existing API conventions before choosing exact mappings.

Expected baseline:

* `201 Created` — order successfully created.
* `400 Bad Request` — malformed or invalid request.
* `401 Unauthorized` — missing or invalid authentication.
* `403 Forbidden` — authenticated user cannot submit the order/customer mismatch.
* `409 Conflict` or `422 Unprocessable Entity` — insufficient inventory, according to existing project conventions.
* `503 Service Unavailable` — inventory service unavailable.
* `504 Gateway Timeout` — inventory availability deadline exceeded.

Do not invent a new error envelope if the platform already provides one.

Map typed adapter failures from ARB-023.3 through the existing application/API error strategy:

* `InventoryAvailabilityTimeoutException`
* `InventoryAvailabilityRemoteUnavailableException`
* `InventoryAvailabilityProtocolException`
* `InventoryAvailabilityInternalException`

Protocol/internal failures must remain distinguishable from normal business unavailability.

### 6. Transactional ordering

Availability verification must happen before order persistence and before any order-created event is emitted.

Required invariant:

```text
Inventory unavailable
    ⇒ no Order saved
    ⇒ no OrderCreated event/outbox entry
    ⇒ no Saga started
```

Do not add inventory reservation in this slice.

`CheckAvailability` remains read-only.

### 7. Tests

Add focused tests at the appropriate layers.

#### REST adapter tests

Cover at least:

* Valid authenticated request returns `201`.
* Invalid payload returns `400`.
* Missing authentication returns `401`.
* Unauthorized customer access returns `403`.
* Inventory unavailable returns the chosen business HTTP status.
* Inventory timeout returns `504`.
* Inventory remote unavailable returns `503`.

#### Application/use-case tests

Cover at least:

* Availability is checked before persistence.
* Fully available request creates and saves an order.
* One unavailable line prevents order creation.
* Unknown SKU/zero availability prevents order creation.
* Multiple lines are checked correctly.
* Inventory timeout prevents persistence.
* Inventory unavailable technical failure prevents persistence.
* No order-created event is emitted on any failed availability check.
* The input collections and returned collections are not mutated unexpectedly.

#### Architecture tests

Confirm:

* REST adapter does not depend on protobuf.
* Application/domain do not depend on gRPC or protobuf.
* REST adapter depends on the application input port, not the outbound gRPC adapter.
* Inventory gRPC implementation remains replaceable behind `InventoryAvailabilityPort`.

#### Integration test

Add at least one production-path integration test covering:

```text
HTTP request
→ REST adapter
→ application use case
→ InventoryAvailabilityPort
→ order persistence
→ HTTP response
```

Use the smallest reliable integration boundary.

Do not create a large multi-service Testcontainers test if the same behavior can be demonstrated with a controlled port stub plus real HTTP and persistence.

Reuse the cross-service gRPC integration coverage already delivered by ARB-023.3 rather than duplicating it unnecessarily.

## Non-Goals

Do not implement:

* Inventory reservation.
* Saga orchestration.
* Kafka consumption.
* Compensation.
* Retry policy.
* Circuit breaker.
* gRPC metadata propagation.
* New authentication architecture.
* Frontend changes.
* API gateway.
* New database migration unless genuinely required by the existing order persistence design.

Do not refactor unrelated order or inventory code.

---

## Architectural Constraints

* Domain and application remain free of Spring, REST, gRPC, and protobuf concerns unless already approved by project architecture.
* Protobuf types remain isolated inside the gRPC adapter.
* REST DTOs remain isolated inside the inbound adapter.
* Order Service owns its application port and application models.
* Inventory Service remains the authority for inventory availability.
* Availability is a precondition, not a reservation.
* No commit.

## Stop Conditions

Stop and report rather than improvising if:

* There is no existing order creation input port or use case.
* Order persistence is not yet implemented.
* The expected REST endpoint conflicts with an existing API contract.
* The error model cannot represent business unavailability without a broader design decision.
* The current use case already creates the order before inventory verification.
* Completing the task would require introducing saga orchestration or reservation semantics.
* Tests expose an architectural contradiction between ARB-007 and ARB-023.3.

At the hard stop, leave the repository in a compilable state when possible and write the checkpoint with completed work, remaining work, failures, and the exact next action.
