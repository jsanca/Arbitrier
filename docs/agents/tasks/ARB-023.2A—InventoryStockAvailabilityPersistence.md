## 1. Intention

Implement the production inventory availability capability required by:

```text
InventoryAvailabilityService.CheckAvailability
```

ARB-023.2 currently exposes a functional gRPC server adapter, application use case, and outbound query port, but the only concrete availability source is an in-memory test adapter.

This task must provide:

* a persistent stock availability model;
* a production implementation of `InventoryAvailabilityQueryPort`;
* a clear definition of available inventory;
* integration between the gRPC server and real persisted stock data;
* end-to-end proof from gRPC request to PostgreSQL query.

The task is complete only when Inventory Service can answer availability queries from persisted production data.

---

## 2. Functional Goal

Given persisted stock such as:

```text
SKU-001
on hand: 10
reserved: 3
available: 7
```

Inventory Service must answer:

```text
request quantity: 7
result: AVAILABLE
```

and:

```text
request quantity: 8
result: UNAVAILABLE
requested: 8
available: 7
reason: INSUFFICIENT_STOCK
```

For an unknown SKU:

```text
result: UNAVAILABLE
reason: PRODUCT_NOT_FOUND
available: 0
```

The response must travel through the complete production path:

```text
gRPC
    ↓
InventoryAvailabilityGrpcService
    ↓
CheckInventoryAvailabilityUseCase
    ↓
InventoryAvailabilityQueryPort
    ↓
production persistence adapter
    ↓
PostgreSQL
```

## 7. Core Domain Decision

The task must define what `availableQuantity` means.

Preferred baseline:

```text
availableQuantity =
    onHandQuantity - reservedQuantity
```

This formula may be used only if it matches the existing reservation semantics.

If reservations are already represented in persisted reservation or allocation tables, determine whether:

```text
reservedQuantity
```

should be:

* stored directly;
* calculated from active reservations;
* calculated from allocations;
* derived through a dedicated query.

Prefer a single source of truth.

Do not maintain two independently mutable representations of reserved stock unless the consistency strategy is explicit and tested.

---

## 8. Recommended Minimal Stock Model

If the current model lacks a suitable stock source, introduce a minimal persistent model.

Suggested table:

```text
inventory_stock
```

Suggested columns:

```text
product_id
on_hand_quantity
version
created_at
updated_at
```

Optional:

```text
warehouse_id
```

only if the current Inventory domain already models stock per warehouse.

Do not add warehouse identity speculatively.

### Suggested semantics

```text
product_id
```

Identifies the inventory-managed SKU.

```text
on_hand_quantity
```

Represents physically or logically owned stock before active reservations are deducted.

```text
version
```

Supports optimistic concurrency for future stock mutations.

The availability query may calculate active reserved quantity from the existing reservation model rather than storing `reserved_quantity` in this table.

## 9. Avoiding Duplicate Sources of Truth

The implementation must explicitly choose between these approaches.

### Option A — Derived reservations

```text
available =
    stock.onHandQuantity
    - sum(active reservation quantities)
```

Advantages:

* reservation tables remain authoritative;
* no duplicated reserved counter;
* easier consistency reasoning.

Risks:

* more complex query;
* possible performance cost.

### Option B — Persisted reserved counter

```text
inventory_stock.reservedQuantity
```

Advantages:

* simple and fast reads.

Risks:

* requires atomic updates with every reservation transition;
* can drift from reservation records;
* expands the task into reservation mutation consistency.

For this slice, prefer **Option A** unless the existing domain already maintains an authoritative reserved counter.

Do not introduce a reserved counter merely to simplify the query.

## 10. Active Reservation Semantics

Define which reservation states consume stock.

For example:

```text
PENDING
CONFIRMED
ALLOCATED
```

may or may not count as active depending on the existing domain model.

Create one explicit application or persistence rule such as:

```text

    [RESERVED, CONFIRMED]
```

Do not scatter state checks across adapters.

If the domain already exposes behavior such as:

```java

```

reuse it.

If the calculation occurs in SQL, centralize and document the state set.

Tests must cover every state that affects availability.

## 11. Persistence Port

Reuse the existing outbound port created in ARB-023.2:

```java
public interface InventoryAvailabilityQueryPort {

    Map<String, InventoryAvailabilitySnapshot> findAvailability(
            Collection<String> productIds
    );
}
```

Do not change its API unless a genuine semantic limitation is found.

The production adapter must:

* query all requested products in a batch;
* avoid one query per SKU;
* return one snapshot per known product;
* omit unknown products from the result;
* never return negative available quantities.

If reservation totals exceed on-hand stock because of legacy or inconsistent data:

```text
availableQuantity = max(0, calculatedQuantity)
```

Log or surface the inconsistency according to existing safe operational conventions.

Do not return a negative availability value through gRPC.

## 15. Query Strategy

The preferred query should retrieve stock and active reservations in one bounded operation.

Possible approaches:

### Aggregate query

```sql
SELECT
    stock.product_id,
    stock.on_hand_quantity,
    COALESCE(SUM(active_line.quantity), 0) AS reserved_quantity
FROM inventory_stock stock
LEFT JOIN ...
WHERE stock.product_id IN (...)
GROUP BY stock.product_id, stock.on_hand_quantity
```

### Two batch queries

```text
query all stock rows
query all active reservation totals
merge in adapter
```

Both are acceptable if:

* the number of queries is constant;
* there is no N+1 behavior;
* semantics remain clear;
* tests prove correctness.

Prefer readability over an unnecessarily clever SQL statement.

## 19. gRPC Exception Mapping Correction

Correct the current mapping:

```text

```

A generic `NullPointerException` may indicate a server programming defect and must not be reported as a client validation error.

Required mapping:

```text
IllegalArgumentException
or explicit validation exception
    → INVALID_ARGUMENT
```

```text
NullPointerException
    → INTERNAL
```

```text
unexpected Exception
    → INTERNAL
```

Add tests proving:

```text

unexpected_null_pointer_maps_to_internal
```

Prefer explicit validation exceptions over relying on generic runtime exception types.

## 28. Guiding Principles

> A service endpoint is not functional merely because it can receive a request; it must have an authoritative answer to return.

> Persisted stock and reservation state must form one coherent model, not two competing truths.

> Availability is derived from current state. Reservation remains the authoritative Saga transaction.

> Software visibility must come from real behavior, not from a hidden test double.
