# ARB-023.3 — Order Inventory Availability gRPC Client Adapter

**Status:** DONE
**Epic:** ARB-023 — Executable Order Entry Flow
**Depends on:** ARB-023.1, ARB-023.2, ARB-023.2A
**Primary module:** `order-service`
**Supporting modules:** `contracts`, `inventory-service` for integration testing
**Owner:** Clio
**Review after completion:** Deep
**Commit policy:** Clio must not commit changes.

---

## 1. Intention

Implement the Order Service outbound gRPC client adapter for querying Inventory availability through:

```text
InventoryAvailabilityService.CheckAvailability
```

The adapter must allow Order application code to ask:

> Does Inventory currently report enough stock to begin processing this prospective order?

The adapter translates an Order application request into the generated protobuf request, invokes Inventory Service synchronously, and maps the protobuf response into an Order-side application result.

This slice must not expose a REST endpoint, create an Order, start a Saga, or reserve inventory.

---

## 2. Functional Goal

Order Service must be able to invoke Inventory Service through the production path:

```text
Order application
    ↓
InventoryAvailabilityPort
    ↓
gRPC client adapter
    ↓
InventoryAvailabilityService
    ↓
Inventory persisted stock
    ↓
AVAILABLE / UNAVAILABLE
```

Example:

```text
Order items:
SKU-001 quantity 7

Inventory:
on hand 10
active allocations 3
available 7

Order-side result:
AVAILABLE
```

And:

```text
Order items:
SKU-001 quantity 8

Order-side result:
UNAVAILABLE
product: SKU-001
requested: 8
available: 7
reason: INSUFFICIENT_STOCK
```

For an unknown SKU:

```text
UNAVAILABLE
reason: PRODUCT_NOT_FOUND
```

## 3. Architectural Flow

Implement:

```text
Order use case
    ↓
```

The gRPC implementation is an outbound adapter.

## 7. Order Application Port

Define an outbound application port owned by Order Service.

Suggested shape:

```java
public interface InventoryAvailabilityPort {

    InventoryAvailabilityResult checkAvailability(
            InventoryAvailabilityRequest request
    );
}
```

The exact name may follow existing Order Service conventions.

The port belongs to Order Service because it expresses what Order needs from Inventory.

Do not reuse the Inventory Service application port directly.

## 9. Order-Side Result Model

The Order application layer must receive its own result model.

Suggested design:

```java
public sealed interface InventoryAvailabilityResult {

    record Available()
            implements InventoryAvailabilityResult {
    }

    record Unavailable(
            List<UnavailableInventoryItem> items
    ) implements InventoryAvailabilityResult {
    }
}
```

Suggested unavailable item:

```java
public record UnavailableInventoryItem(
        String productId,
        int requestedQuantity,
        int availableQuantity,
        InventoryUnavailabilityReason reason
```

Suggested enum:

```java
public enum InventoryUnavailabilityReason {
    INSUFFICIENT_STOCK,
    PRODUCT_NOT_FOUND

```

Do not expose protobuf enums outside the gRPC adapter.

## 10. Ownership of Validation

The Order application boundary should validate structural requirements before invoking gRPC:

```text
non-blank requestId
non-empty items
non-blank productId
quantity > 0
no duplicate productId
```

However, Inventory remains the authoritative validator for its contract.

The client adapter must not assume that local validation eliminates the need to handle:

```text
INVALID_ARGUMENT
```

from the remote service.

Avoid duplicating large validation frameworks solely for this slice.

## 11. gRPC Client Adapter

Suggested class:

```java
public final class GrpcInventoryAvailabilityAdapter
        implements InventoryAvailabilityPort {
}
```

Suggested logical placement:

```text
order-service
└── adapter
    └── outbound
        └── grpc
            └── inventory
                ├── GrpcInventoryAvailabilityAdapter
                ├── InventoryAvailabilityGrpcRequestMapper
                ├── InventoryAvailabilityGrpcResponseMapper
                └── InventoryAvailabilityGrpcExceptionMapper
```

The main adapter method should remain visually simple:

```text
map application request
        ↓
apply deadline
        ↓
invoke generated stub
        ↓
map protobuf response
        ↓
return application result
```

Do not put Order business decisions inside the adapter.

## 12. Stub Selection

Prefer the generated blocking stub for this slice unless the current Order application model is already reactive or asynchronous.

Suggested invocation:

```java
blockingStub
    .withDeadlineAfter(timeout)
    .checkAvailability(request);
```

A blocking stub is acceptable because:

* the RPC is unary;
* the first executable flow is intentionally simple;
* the application is not currently modeled as reactive;
* Java virtual threads can later absorb blocking I/O efficiently where configured.

Do not introduce Reactor, CompletableFuture orchestration, or custom thread pools merely for this call.

Document the selected stub and why.

## 13. Request Mapping

Map:

```text
InventoryAvailabilityRequest
```

to:

```text
CheckAvailabilityRequest
```

Rules:

```text
requestId
    → request_id

items[].productId
    → items[].product_id

items[].quantity
    → items[].quantity
```

Every requested Order item must appear exactly once.

Do not aggregate duplicate products inside the adapter.

Duplicates should be rejected before invocation.

---

## 14. Response Mapping

### Available

Map:

```text
AVAILABILITY_STATUS_AVAILABLE
```

to:

```java
new InventoryAvailabilityResult.Available()
```

The protobuf response must not contain unavailable items in this state.

If it does, treat it as an invalid remote response.

### Unavailable

Map:

```text
AVAILABILITY_STATUS_UNAVAILABLE
```

to:

```java
new InventoryAvailabilityResult.Unavailable(...)
```

Map every unavailable item.

Reason mapping:

```text
UNAVAILABILITY_REASON_INSUFFICIENT_STOCK
    → INSUFFICIENT_STOCK

UNAVAILABILITY_REASON_PRODUCT_NOT_FOUND
    → PRODUCT_NOT_FOUND
```

### Unspecified status

If Inventory returns:

```text
AVAILABILITY_STATUS_UNSPECIFIED
```

the client must treat it as a technical protocol failure.

Do not convert it into business unavailability.

---

## 15. Remote Response Validation

Validate remote response invariants.

### AVAILABLE

Required:

```text
status = AVAILABLE
unavailable_items = empty
```

### UNAVAILABLE

Required:

```text
status = UNAVAILABLE
unavailable_items is not empty
```

Each unavailable item must contain:

```text
non-blank product_id
requested_quantity > 0
available_quantity >= 0
known reason
```

An invalid response must raise a technical integration exception.

Do not let malformed remote data silently enter Order application logic.

---

## 16. Technical Error Model

Do not expose `StatusRuntimeException` to the Order application layer.

Introduce or reuse a technical exception hierarchy.

Possible shape:

```java
public class InventoryAvailabilityException
        extends RuntimeException {
}
```

Possible specialized errors:

```text
InventoryAvailabilityInvalidRequestException
InventoryAvailabilityUnavailableException
InventoryAvailabilityTimeoutException
InventoryAvailabilityProtocolException
InventoryAvailabilityInternalException
```

Keep the hierarchy as small as practical.

At minimum, distinguish:

```text
client/input error
remote unavailable
deadline exceeded
remote/internal failure
protocol violation
```

Do not represent technical failures as:

```text
InventoryAvailabilityResult.Unavailable
```

Business unavailability and communication failure are different outcomes.

---

## 17. gRPC Status Mapping

Suggested mapping:

```text
INVALID_ARGUMENT
    → explicit client/integration request exception

DEADLINE_EXCEEDED
    → timeout exception

UNAVAILABLE
    → remote service unavailable exception

CANCELLED
    → communication exception

INTERNAL
UNKNOWN
DATA_LOSS
    → remote internal exception

UNAUTHENTICATED
PERMISSION_DENIED
    → remote access exception
```

Other statuses may map to a generic inventory communication exception.

Preserve the original cause where safe.

Do not expose remote stack traces or raw server descriptions through public Order responses in future slices.

## 18. Deadline

Every gRPC invocation must have a finite deadline.

Suggested configuration:

```yaml
arbitrier:
  order:
    inventory:
      grpc:
        target: ${INVENTORY_GRPC_TARGET:static://localhost:9091}
        deadline: ${INVENTORY_GRPC_DEADLINE:2s}
```

Follow the selected gRPC integration library’s actual configuration format.

Do not hardcode the timeout inside the adapter.

Recommended default:

```text
2 seconds
```

The exact value may be adjusted if project conventions already exist.

No retry should be added in this slice.

A timeout must fail clearly rather than allow indefinite blocking.

## 31. Transaction Semantics

The client invocation must not run inside a database transaction unless the caller already owns one for a documented reason.

This slice itself must not add:

```java
@Transactional
```

around the remote call.

Future Order entry logic should avoid holding a local database transaction open while waiting on Inventory.

No distributed transaction is introduced.

---

## 32. Concurrency Semantics

The client returns a point-in-time Inventory observation.

Order Service must not infer:

```text
AVAILABLE
    =
stock guaranteed for future reservation
```

The result means:

```text
Inventory currently reports enough availability
to justify beginning the transaction.
```

The later Saga reservation remains authoritative.

Document this explicitly in Order Service.

---

## 33. Behavior on Technical Failure

```text
504
409
422
```

That belongs to the inbound Order flow.

The client adapter should provide a precise technical failure to its caller.

Do not create a fake `Unavailable` business result when the Inventory service cannot be reached.

## 39. Guiding Principles

> The consumer owns the port; the provider owns the contract implementation.

> A remote business rejection is data. A communication failure is an exception.

> Availability justifies beginning the transaction; it does not guarantee its success.

> A client adapter should make the remote service look like an application capability, not leak the transport into the business.
