# ARB-023.1 — Inventory Availability gRPC Contract

**Status:** DONE
**Epic:** ARB-023 — Executable Order Entry Flow
**Type:** Contract Slice
**Primary module:** `contracts`
**Owner:** Clio
**Commit policy:** Clio must not commit changes.

---

## 1. Intention

Define the synchronous gRPC contract that allows Order Service to query Inventory Service for the current availability of all items requested in an order.

This operation is a read-only business precondition used before accepting an order and starting its Saga.

The contract answers:

> Does Inventory currently report enough available stock to begin processing this order?

The operation does not reserve, allocate, decrement, lock, or otherwise mutate inventory.

Actual inventory reservation remains exclusively part of the Saga transaction flow.

---

## 6. Service Definition

Define a unary gRPC service:

```proto
service InventoryAvailabilityService {
  rpc CheckAvailability(CheckAvailabilityRequest)
      returns (CheckAvailabilityResponse);
}
```

The RPC represents a synchronous query.

It must not imply reservation or mutation.

The method name should remain `CheckAvailability` unless existing naming conventions require a clearly superior equivalent.

---

## 7. Request Model

Define:

```proto
message CheckAvailabilityRequest {
  string request_id = 1;
  repeated RequestedItem items = 2;
}
```

### 7.1 `request_id`

Purpose:

* correlation;
* traceability;
* diagnostic association;
* future idempotency support at the caller boundary.

This is not an inventory reservation identifier.

It must not be interpreted as an idempotency guarantee in this slice.

### 7.2 `items`

Represents all products and quantities required by the prospective order.

The request must support more than one item.

Define:

```proto
message RequestedItem {
  string product_id = 1;
  int32 quantity = 2;
}
```

### 7.3 Field semantics

#### `product_id`

* required by application semantics;
* must identify one inventory-managed product;
* must not be blank;
* remains a string at the wire boundary;
* no domain-specific Java type belongs in protobuf.

#### `quantity`

* required by application semantics;
* must be greater than zero;
* represents the quantity requested by the order;
* must not represent reserved quantity.

Protocol Buffers does not enforce these business validations by itself. Document the invariants in comments and enforce them later in the adapters/application layer.

---

## 8. Response Model

Define:

```proto
message CheckAvailabilityResponse {
  AvailabilityStatus status = 1;
  repeated UnavailableItem unavailable_items = 2;
}
```

The result must allow the caller to distinguish:

* all items currently available;
* one or more items currently unavailable;
* unspecified or invalid protocol state.

---

## 9. Availability Status

Define:

```proto
enum AvailabilityStatus {
  AVAILABILITY_STATUS_UNSPECIFIED = 0;
  AVAILABILITY_STATUS_AVAILABLE = 1;
  AVAILABILITY_STATUS_UNAVAILABLE = 2;
}
```

Use prefixed enum constants to avoid generated-code name collisions and make wire semantics explicit.

### Semantics

#### `AVAILABILITY_STATUS_AVAILABLE`

Every requested item currently has sufficient available stock.

Expected response:

```text
unavailable_items = empty
```

#### `AVAILABILITY_STATUS_UNAVAILABLE`

At least one requested item does not currently have sufficient available stock.

Expected response:

```text
unavailable_items = one or more entries
```

#### `AVAILABILITY_STATUS_UNSPECIFIED`

Represents an absent, unknown, invalid, or forward-compatibility state.

Application adapters must not treat this value as availability.

---

## 10. Unavailable Item Details

Define:

```proto
message UnavailableItem {
  string product_id = 1;
  int32 requested_quantity = 2;
  int32 available_quantity = 3;
  UnavailabilityReason reason = 4;
}
```

Define:

```proto
enum UnavailabilityReason {
  UNAVAILABILITY_REASON_UNSPECIFIED = 0;
  UNAVAILABILITY_REASON_INSUFFICIENT_STOCK = 1;
  UNAVAILABILITY_REASON_PRODUCT_NOT_FOUND = 2;
}
```

### Semantics

#### `requested_quantity`

The quantity originally requested.

#### `available_quantity`

The quantity currently available according to Inventory Service.

For an unknown product, this may be zero.

#### `reason`

Allows Order Service and future REST adapters to distinguish:

* insufficient stock;
* unknown product;
* unspecified response.

Do not add operational or infrastructure failures to this enum.

Failures such as timeout, unavailable service, malformed response, or internal server error belong to gRPC status/error handling, not to the business response.

## 11. Complete Baseline Contract

The resulting contract should be semantically equivalent to:

```proto
syntax = "proto3";

package arbitrier.inventory.v1;

option java_multiple_files = true;
option java_package = "io.arbitrier.contracts.inventory.v1";
option java_outer_classname = "InventoryAvailabilityProto";

service InventoryAvailabilityService {
  rpc CheckAvailability(CheckAvailabilityRequest)
      returns (CheckAvailabilityResponse);
}

message CheckAvailabilityRequest {
  string request_id = 1;
  repeated RequestedItem items = 2;
}

message RequestedItem {
  string product_id = 1;
  int32 quantity = 2;
}

message CheckAvailabilityResponse {
  AvailabilityStatus status = 1;
  repeated UnavailableItem unavailable_items = 2;
}

message UnavailableItem {
  string product_id = 1;
  int32 requested_quantity = 2;
  int32 available_quantity = 3;
  UnavailabilityReason reason = 4;
}

enum AvailabilityStatus {
  AVAILABILITY_STATUS_UNSPECIFIED = 0;
  AVAILABILITY_STATUS_AVAILABLE = 1;
  AVAILABILITY_STATUS_UNAVAILABLE = 2;
}

enum UnavailabilityReason {
  UNAVAILABILITY_REASON_UNSPECIFIED = 0;
  UNAVAILABILITY_REASON_INSUFFICIENT_STOCK = 1;
  UNAVAILABILITY_REASON_PRODUCT_NOT_FOUND = 2;
}
```

Clio may adjust package names and file paths to match established repository conventions.

Clio must not broaden the business scope.

## 12. Contract Invariants

Document these invariants in the `.proto` file:

### Request invariants

```text
request_id must not be blank
items must contain at least one entry
product_id must not be blank
quantity must be greater than zero
product_id must not appear more than once
```

Duplicate product IDs should be rejected by the future adapter rather than silently aggregated.

This keeps the wire request unambiguous.

### Response invariants

For `AVAILABLE`:

```text
unavailable_items must be empty
```

For `UNAVAILABLE`:

```text
unavailable_items must not be empty
```

Every `UnavailableItem` must correspond to a product present in the request.

`available_quantity` must not be negative.

---

## 13. Query Semantics

Availability is a point-in-time observation.

A successful result means:

> Inventory Service currently reports enough available stock.

It does not guarantee that stock will still be available when the Saga attempts reservation.

Therefore, the following remains valid:

```text
CheckAvailability = AVAILABLE
        ↓
another transaction consumes stock
        ↓
Saga reservation = REJECTED
```

This is expected distributed-system behavior, not an inconsistency in the contract.

The Saga remains the authoritative mechanism for inventory reservation.

---

## 14. Failure Semantics

Business unavailability must be returned through:

```text
CheckAvailabilityResponse.status = UNAVAILABLE
```

Technical failures must use gRPC status errors.

Examples:

```text
INVALID_ARGUMENT
```

For structurally or semantically invalid requests.

```text
UNAVAILABLE
```

When Inventory Service cannot currently serve the request.

```text
INTERNAL
```

For an unexpected server failure.

Do not encode technical failures inside `AvailabilityStatus`.

This slice defines the distinction but does not implement error mapping.
