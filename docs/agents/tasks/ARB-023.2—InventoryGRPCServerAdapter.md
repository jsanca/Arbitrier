# ARB-023.2 — Inventory gRPC Server Adapter

**Status:** DONE
**Epic:** ARB-023 — Executable Order Entry Flow
**Depends on:** ARB-023.1 — Inventory Availability gRPC Contract
**Primary module:** `inventory-service`
**Owner:** Clio
**Commit policy:** Clio must not commit changes.

---

## 1. Intention

Implement the Inventory Service gRPC server adapter for:

```text
InventoryAvailabilityService.CheckAvailability
```

The adapter must receive a synchronous availability request, validate and translate its protobuf payload, invoke an Inventory application query, and map the application result back to the generated protobuf response.

This slice turns the ARB-023.1 wire contract into an executable Inventory Service capability.

The operation is read-only.

It must not reserve, allocate, decrement, lock, claim, or mutate inventory.

---

## 2. Business Question

The operation answers:

> Does Inventory currently report enough available stock for every item requested by the prospective order?

A positive response means only that the precondition is currently satisfied.

It does not guarantee that stock will remain available when the Saga later attempts the authoritative reservation.

The following sequence remains valid:

```text
CheckAvailability → AVAILABLE
        ↓
stock changes
        ↓
Saga reservation → REJECTED
```

That is expected distributed-system behavior.

## 9. Query Semantics

The use case must evaluate every requested product.

For each item:

### Product exists and stock is sufficient

```text
availableQuantity >= requestedQuantity
```

The item is available.

### Product exists but stock is insufficient

Return:

```text
reason = INSUFFICIENT_STOCK
requestedQuantity = request quantity
availableQuantity = current available quantity
```

### Product does not exist

Return:

```text
reason = PRODUCT_NOT_FOUND
requestedQuantity = request quantity
availableQuantity = 0
```

### Overall result

Return `Available` only when every requested item is available.

Return `Unavailable` when one or more requested items fail the precondition.

All unavailable items must be included in the result. Do not stop at the first failure.

This makes the response useful to callers and avoids multiple round trips.

---

## 10. Inventory Quantity Definition

Use the existing Inventory domain definition of stock available for reservation.

Do not invent a new calculation in the adapter.

The implementation must identify what the current model considers available inventory.

For example, if the existing model distinguishes:

```text
on-hand
reserved
available
```

the use case must use the established domain rule:

```text
available = on-hand - reserved
```

only if that is already the domain definition.

If the model exposes an existing method such as:

```java
inventory.availableQuantity()
```

prefer that method.

The gRPC adapter must never reproduce this calculation.

---

## 11. Query Port

Reuse an existing Inventory repository or query port if it can retrieve the information without violating architectural boundaries.

If a new port is required, keep it specific to the application need.

Possible shape:

```java
public interface InventoryAvailabilityQueryPort {

    Map<ProductId, InventoryAvailabilitySnapshot> findAvailability(
            Collection<ProductId> productIds
    );
}
```

Possible snapshot:

```java
public record InventoryAvailabilitySnapshot(
        ProductId productId,
        int availableQuantity
) {
}
```

Absence from the returned result represents `PRODUCT_NOT_FOUND`.

Avoid an N+1 query if the existing persistence technology can retrieve all requested products in one operation.

However, do not introduce premature persistence complexity if the current in-memory or repository model naturally supports individual lookup.

Correctness and architectural clarity are the primary goals of this slice.

## 12. Request Validation

Validate the application semantics before executing the query.

Reject:

```text
blank request_id
empty items
blank product_id
quantity <= 0
duplicate product_id
```

Duplicate product IDs must not be silently merged.

The protobuf contract describes a canonical request where each product appears once.

The adapter may delegate validation to an application command/query factory or validator if the project already has established validation infrastructure.

Do not place all validation directly inside the gRPC method if a reusable application boundary exists.

## 14. Protobuf Request Mapping

Map:

```text
CheckAvailabilityRequest
```

to the application query.

Mapping rules:

```text
request_id → requestId
items[].product_id → ProductId
items[].quantity → requested quantity
```

No generated protobuf class may cross into the domain or application service.

---

## 15. Protobuf Response Mapping

### Available

Map:

```java
InventoryAvailabilityResult.Available
```

to:

```text
status = AVAILABILITY_STATUS_AVAILABLE
unavailable_items = empty
```

### Unavailable

Map:

```java
InventoryAvailabilityResult.Unavailable
```

to:

```text
status = AVAILABILITY_STATUS_UNAVAILABLE
unavailable_items = all unavailable items
```

Reason mapping:

```text
INSUFFICIENT_STOCK
    →
UNAVAILABILITY_REASON_INSUFFICIENT_STOCK

PRODUCT_NOT_FOUND
    →
UNAVAILABILITY_REASON_PRODUCT_NOT_FOUND
```

The adapter must never return:

```text
AVAILABILITY_STATUS_UNSPECIFIED
```

for a successful application invocation.

`UNSPECIFIED` is a protocol default, not a valid successful business result.

---

## 16. Error Mapping

Keep business unavailability separate from technical failure.

### Invalid request

Map validation failures to:

```text
Status.INVALID_ARGUMENT
```

Include a safe, concise description.

Do not expose stack traces or internal persistence details.

### Unexpected application or infrastructure failure

Map to:

```text
Status.INTERNAL
```

or the project’s established equivalent.

### Temporarily unavailable dependency

If the Inventory query infrastructure explicitly reports a temporary availability failure, map it to:

```text
Status.UNAVAILABLE
```

Do not fabricate this distinction if the current application model does not support it.

### Business unavailability

Never map insufficient stock or missing products to a gRPC error.

Those are successful RPC responses with:

```text
AVAILABILITY_STATUS_UNAVAILABLE
```

---

## 17. StreamObserver Safety

The adapter must complete each successful request with exactly:

```java
responseObserver.onNext(response);
responseObserver.onCompleted();
```

On failure, invoke:

```java
responseObserver.onError(statusException);
```

Do not call `onCompleted()` after `onError()`.

Do not emit more than one response for this unary method.

## 20. Transaction Semantics

This operation is a read-only query.

If a transaction is required by the persistence adapter, use a read-only transaction where supported:

```java

```

## 21. Concurrency Semantics

The query returns a point-in-time observation.

It does not acquire locks intended to preserve availability for a future order.

Avoid:

```text
SELECT ... FOR UPDATE
```

unless the existing Inventory read model already requires it for a separate, documented reason.

The Saga reservation remains responsible for authoritative concurrency control.

## 31. Guiding Principles

> A transport adapter translates and delegates; it does not become the application.

> Availability is an observation. Reservation is a transaction.

> The first executable service interaction should make the architecture easier to understand, not introduce another hidden subsystem.
